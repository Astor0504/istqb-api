01 單元6：驗收測試與測試類型

02 學習目標

  • 理解驗收測試的目的與執行方式
  • 掌握四種驗收測試類型
  • 區分功能性測試與非功能性測試
  • 了解黑箱、白箱測試的差異
  • 理解確認性測試與迴歸測試

03 6.1 驗收測試（Acceptance Testing）⭐⭐⭐

定義

從使用者角度驗證系統是否滿足業務需求

核心目的

不是：找 Bug 而是：建立信心

說明：

  • 大部分 Bug 應在系統測試階段發現
  • 驗收測試發現太多 Bug，使用者會不滿
  • 目的是增加使用者對系統的信賴感

執行者

理想：最終使用者

原因：

  • 最了解業務需求
  • 知道實際使用情境
  • 能判斷是否符合期望

測試時機

時間點：

  • 上線前
  • 可執行多次
  • 直到使用者滿意

四種驗收測試類型 ⭐⭐⭐ 必背

04 6.2 使用者驗收測試（UAT）

User Acceptance Testing

定義

驗證系統滿足業務需求與期望

核心特性

1. 多方觀點

  • 不只一個角色
  • 不同使用者視角
  • 全面性驗證

例子：物流系統

司機視角：
  - 路線規劃是否清楚
  - 操作是否方便

店員視角：
  - 收貨流程是否順暢
  - 庫存是否正確更新

管理者視角：
  - 報表是否完整
  - 資料是否準確

2. 真實環境

  • 盡量在實際環境測試
  • 使用真實資料
  • 模擬實際作業流程

3. 業務場景

  • 測試完整業務流程
  • 端到端測試
  • 確保符合業務目標

執行方式

時機：

  • 系統測試完成後
  • 上線前的最後驗證

頻率：

  • 可執行多次
  • 直到通過為止

05 6.3 運行驗收測試（Operational Acceptance Testing）

定義

確保系統在生產環境中具備穩定運行條件

測試重點

1. 維護性

  • 系統是否容易維護
  • 更新是否順暢
  • 問題是否容易排除

2. 備份與還原

  • 備份機制是否正常
  • 資料還原是否完整
  • 災難復原是否可行

3. 使用者管理

  • 新增使用者
  • 刪除使用者
  • 權限管理

4. 安全程序

  • 定期安全檢查
  • 漏洞修補流程
  • 存取控制

5. 災難復原

  • 系統故障處理
  • 緊急應變計畫
  • 復原時間目標

現代 vs 傳統

傳統環境（以前）

特性：

  • 軟體安裝在本地
  • 需要實體媒體（磁碟片、光碟）
  • 維護由內部人員負責

重視項目：

  • 安裝流程
  • 系統維護
  • 備份還原
  • 使用者新增刪除

例子：5¼ 吋磁碟片時代

安裝軟體需要：
  磁碟片1 → 插入、安裝
  磁碟片2 → 抽換、繼續
  磁碟片3 → 抽換、完成

現代環境（現在）

特性：

  • 雲端服務
  • 網路應用
  • 自動更新

重視項目變化：

  • 備份還原：雲端自動處理
  • 使用者管理：線上自助
  • 系統維護：服務商負責

結論：

  • 傳統環境：運行驗收測試很重要
  • 現代環境：重要性降低（但仍需要）

06 6.4 合約與法規測試

Contractual and Regulatory Acceptance Testing

定義

驗證系統符合合約要求、產業標準或法律法規

測試內容

1. 合約要求

例子：

  • 系統回應時間需在 3 秒內
  • 資料準確度需達 99.9%
  • 可用性需達 99.5%

特性：

  • 白紙黑字
  • 明確標準
  • 強制遵守

2. 產業標準

常見標準：

  • PCI DSS：金流資料安全標準
  • CNA：資安認證
  • ISO 27001：資訊安全管理

特性：

  • 非強制（但建議遵守）
  • 提升競爭力
  • 客戶信任度

3. 法律法規

例子：

  • GDPR（歐盟）：個人資料保護
  • HIPAA（美國）：醫療資訊隱私
  • 個資法（台灣）：個人資料保護

特性：

  • 強制遵守
  • 違反有罰則
  • 不符合無法上市

4. 功能安全標準（車載通）

例子：ISO 26262

  • 定義 ASIL 等級（A, B, C, D）
  • 要求特定覆蓋率
  • 必須符合才能上市

要求：

ASIL B：需達到分支覆蓋
ASIL C：需達到 MC/DC 覆蓋
ASIL D：更嚴格要求

07 6.5 Alpha/Beta 測試

Alpha 測試

定義

在開發公司環境進行的測試

特性

環境：

  • 公司內部
  • 模擬的使用者環境
  • 受控環境

執行者：

  • 公司內部人員
  • 模擬使用者
  • 潛在客戶

目的：

  • 早期發現問題
  • 在受控環境下測試
  • 降低風險

優點：

  • 問題容易追蹤
  • 溝通方便
  • 修復快速

缺點：

  • 環境不夠真實
  • 無法模擬所有情境
  • 使用者行為受限

Beta 測試

定義

在真實使用者環境進行的測試

特性

環境：

  • 使用者真實環境
  • 各種設備配備
  • 不受控環境

執行者：

  • 真實使用者
  • 志願測試者
  • 早期採用者

目的：

  • 在真實環境驗證
  • 收集使用者回饋
  • 發現環境相容問題

優點：

  • 環境真實多樣
  • 使用者回饋直接
  • 發現實際問題

缺點：

  • 問題難以重現
  • 溝通成本高
  • 修復週期長

經典案例：Microsoft Office 2010

事件背景

策略：

  • 搶先版 99 元
  • 使用期限一年
  • 完整功能

爭議：

  • 使用者付費
  • 卻幫忙測試（發現 Bug）
  • 回傳錯誤資訊給 Microsoft

討論：

到底是「我付 99 元幫你測試」 還是「你付我 99 元讓我測試」？

現代 Beta 測試

正確做法：

  • 免費讓使用者測試
  • 明確告知是測試版
  • 使用者自願參與

例子：遊戲封測

  • 完全免費
  • 使用者樂意參與
  • 回饋寶貴意見

Beta 測試的價值

為何需要：

使用者環境千變萬化，公司無法完全模擬

使用者環境多樣性：

作業系統：Windows 7/8/10/11, macOS, Linux...
瀏覽器：Chrome, Firefox, Safari, Edge...
螢幕：1024x768, 1920x1080, 4K, 多螢幕...
網路：有線、WiFi、4G/5G、慢速...
硬體：各種 CPU、記憶體、顯示卡組合...

公司無法：

  • 準備所有設備組合
  • 測試所有可能情境
  • 預測所有使用方式

解決方案：

  • 讓使用者測試
  • 真實環境回饋
  • 發現相容性問題

Alpha vs Beta 比較表

項目Alpha 測試Beta 測試
環境公司內部使用者環境
執行者內部人員真實使用者
控制程度高低
費用公司成本應免費提供
問題追蹤容易困難
環境真實性低高
修復速度快慢
記憶口訣：
  • Alpha：A = At company（在公司）
  • Beta：B = By users（由使用者）

08 6.6 測試類型（Test Types）

四種基本測試類型 ⭐⭐⭐ 必考

官方分類：

  1. 功能性測試（Functional Testing）
  2. 非功能性測試（Non-functional Testing）
  3. 黑箱測試（Black-box Testing）
  4. 白箱測試（White-box Testing）

重要觀念：

測試類型適用於所有測試層級

意思：

  • 單元測試可做功能性測試
  • 單元測試也可做效能測試
  • 整合測試可做安全測試
  • 不限於特定層級

09 6.7 功能性測試

Functional Testing

定義

測試系統是否正確執行應有的功能

重點

關注：功能有沒有 不關注：功能好不好

例子：

✓ 有查詢功能嗎？
✓ 有新增功能嗎？
✓ 有計算功能嗎？
✓ 計算結果正確嗎？

✗ 查詢速度快嗎？（這是非功能性）
✗ 介面美觀嗎？（這是非功能性）

測試依據

主要來源：功能需求

可在以下找到：

  • 使用案例（Use Case）
  • 規格書（Specification）
  • 使用者故事（User Story）
  • 業務流程（Business Process）

領域知識的重要性 💡

案例：股票軟體測試

背景：

  • 測試工程師剛畢業
  • 分配到金融公司
  • 測試股票軟體

問題：

  • 從未買過股票
  • 不懂股票交易
  • 看不懂股票軟體

解決：

  • 公司送他去上課 3 個月
  • 學習股票交易知識
  • 才能進行測試

啟示：

測試人員需要具備領域知識

為什麼：

  • 不懂業務，不知如何測試
  • 無法設計有效測試案例
  • 容易遺漏關鍵情境

10 6.8 非功能性測試

Non-functional Testing

定義

測試系統的品質屬性（How well）

核心概念

功能性：做什麼（What） 非功能性：做得如何（How well）

關鍵字：程度

ISO 25010 品質特徵 ⭐⭐⭐ 必背

2023 年版本，共 8 個品質特徵（v4.0.1）：

1. 功能適用性（Functional Suitability）

說明：功能的完整性與適用性

子特徵：

  • 功能完整性
  • 功能正確性
  • 功能適切性

2. 效能效率（Performance Efficiency）

說明：系統效能與資源使用

包含：

  • 時間行為（速度）
  • 資源使用
  • 容量

例子：

✓ 首頁 5 秒內載入
✓ 同時支援 1000 人在線
✓ CPU 使用率低於 80%

Google 2017 調查：

網頁首頁 5 秒內未完整顯示 使用者可能不再訪問

3. 相容性（Compatibility）

說明：與其他系統共存或交換資料

包含：

  • 共存性
  • 互通性

例子：

✓ 支援 iOS、Android
✓ 支援 Chrome、Safari、Firefox
✓ 與現有系統整合

4. 易用性（Usability）

說明：使用者操作的便利性

包含：

  • 可辨識性
  • 易學性
  • 易操作性
  • 使用者錯誤防護
  • 使用者介面美觀

例子：

✓ 介面直覺
✓ 容易上手
✓ 錯誤訊息清楚
✓ 符合使用習慣

5. 可靠性（Reliability）

說明：系統持續正常運作的能力

包含：

  • 成熟度
  • 可用性
  • 容錯性
  • 可恢復性

例子：

✓ 不易當機
✓ 錯誤處理得當
✓ 故障能快速恢復

6. 安全性（Security）

說明：保護資料與系統安全

包含：

  • 機密性
  • 完整性
  • 不可否認性
  • 可歸責性
  • 真實性

例子：

✓ 密碼加密
✓ 防止 SQL Injection
✓ 存取控制
✓ 稽核紀錄

7. 可維護性（Maintainability）

說明：系統易於維護與修改

包含：

  • 模組化
  • 可重用性
  • 可分析性
  • 可修改性
  • 可測試性

例子：

✓ 程式碼易讀
✓ 模組化設計
✓ 容易除錯
✓ 容易擴充

8. 可移植性（Portability）

說明：系統轉移到不同環境的能力

包含：

  • 適應性
  • 可安裝性
  • 可替換性

例子：

✓ 跨平台運作
✓ 容易安裝
✓ 容易移轉

9. 靈活性（Flexibility）

說明：2023 年新增

說明：系統適應變更的能力

記憶技巧 🎯

口訣：功效相易靠安維移靈

分組記憶：

功能組：功能適用性
效能組：效能效率、相容性
使用組：易用性、可靠性
安全組：安全性
維護組：可維護性、可移植性、靈活性

考試重點：

  • 只需記 8 個品質特徵（v4.0.1）名稱
  • 進階題才需背 35 個子特徵

非功能性測試的重要性

現代競爭重點

功能性容易達成，非功能性才是差異

例子：電商網站

功能性：
  - 每家都有查詢、購物車、結帳
  - 差異不大

非功能性：
  - 速度快慢（影響使用者體驗）
  - 安全程度（影響信任度）
  - 介面美觀（影響使用意願）
  → 這才是勝負關鍵

發現時機的影響

越晚發現，風險越高

例子：網頁載入速度

需求階段：寫「5 秒」
實際需求：應該是「0.5 秒」
  ↓
系統測試才發現
  ↓
要從 5 秒優化到 0.5 秒
  ↓
成本極高！

結論：

  • 非功能性需求要寫明確
  • 早期確認規格
  • 避免後期大幅修改

11 6.9 功能性 vs 非功能性的區分

區分技巧

判斷方法

關鍵字：程度

有程度 → 非功能性 無程度 → 功能性

例子分析

例1：「有查詢功能嗎？」

  • 有/無：功能性 ✓

例2：「查詢速度快嗎？」

  • 快/慢（程度）：非功能性 ✓

例3：「介面美觀嗎？」

  • 美/醜（程度）：非功能性 ✓

例4：「查詢功能必須在 5 秒內回應」

  • 同時包含功能性與非功能性
  • 可視為功能性的延伸

模糊地帶

情境：口罩預約系統

功能：可以點擊預約按鈕 ✓（功能性）
問題：點擊後一直轉圈 ✗（非功能性）

分析：

  • 功能存在（按鈕可按）
  • 但效能不佳（轉圈太久）

測試環境考量

非功能性測試的特殊需求

1. 可用性測試

  • 需要實驗室
  • 透明鏡觀察
  • 記錄使用者行為

2. 效能測試

  • 需要效能測試工具
  • 特定硬體環境
  • 監控系統資源

3. 安全測試

  • 需要滲透測試工具
  • 隔離的測試環境
  • 專業安全知識

12 6.10 黑箱測試與白箱測試

黑箱測試（Black-box Testing）

定義

不看程式碼，只看輸入輸出

特性

1. 專注外部行為

  • 系統界面
  • 輸入輸出
  • 功能表現

2. 測試案例獨立於內部結構

  • 程式碼怎麼寫不重要
  • 只看符不符合規格

3. 不需要閱讀程式碼能力

  • 測試人員不必懂程式
  • 依據規格設計測試

4. 高度依賴文件

  • 規格書
  • 使用者故事
  • 需求文件

適用範圍

  • 功能性測試 ✓
  • 非功能性測試 ✓
  • 各種測試層級 ✓

常見技術（明天詳細教）

  • 等價劃分
  • 邊界值分析
  • 決策表測試
  • 狀態轉換測試

白箱測試（White-box Testing）

定義

關注程式碼內部結構

特性

1. 專注內部結構

  • 程式碼
  • 分支
  • 路徑
  • 資料流

2. 容易自動化

  • 分析程式碼結構
  • 工具支援度高

3. 可計算覆蓋率

  • 敘述覆蓋率
  • 分支覆蓋率
  • 路徑覆蓋率

4. 需要程式設計知識

  • 要看得懂程式碼
  • 理解邏輯結構

主要用途

  • 單元測試（最常用）
  • 元件整合測試
  • 程式碼審查

常見技術（明天詳細教）

  • 敘述測試（Statement Testing）
  • 分支測試（Branch Testing）
  • 其他 5 種（進階）

黑箱 vs 白箱比較

項目黑箱測試白箱測試
關注點外部行為內部結構
依據規格文件程式碼
技能需求領域知識程式設計
覆蓋率可計算可計算
自動化較難較易
主要層級系統、驗收單元、整合

13 6.11 確認性測試與迴歸測試

確認性測試（Confirmation Testing）

定義

重測已修復的缺陷

核心概念

簡單說：重測

目的：確認原始缺陷已成功修復

執行方式

方式1：重新執行失敗的測試案例

Bug 報告：測試案例 A 失敗
  ↓ 開發修復

範圍

聚焦：

  • 只測有問題的地方
  • 不測其他地方

優點：

  • 節省時間
  • 快速驗證

缺點：

  • 可能遺漏連鎖影響
  • 無法發現新問題

迴歸測試（Regression Testing）

定義

確保修改不會影響原有功能

什麼是「迴歸」？

迴歸（Regression）：

無意識的副作用

例子：

修改功能 A
  ↓ 沒注意到
影響到功能 B（原本正常）
  ↓
功能 B 變成不正常 ← 這就是迴歸

為何會發生迴歸？

原因：

  1. 程式碼耦合度高
  2. 共用變數或函式
  3. 修改時沒注意影響範圍

比喻：牽一髮而動全身

執行方式

理想做法：

修改功能 A
  ↓ 分析影響
發現可能影響 B、C
  ↓ 執行
測試 A（確認性）
測試 B（迴歸）
測試 C（迴歸）

全部重測 vs 選擇性重測：

方式優點缺點
全部重測不會遺漏耗時、成本高
選擇性重測節省時間可能遺漏

建議：

  • 有可追溯性：選擇性重測
  • 無可追溯性：全部重測
  • 核心功能：一定要測

雙向可追溯性的重要

有可追溯性：

修改需求 R1
  ↓ 查詢
R1 對應測試案例：TC1, TC3
  ↓ 執行
只測 TC1, TC3（節省時間）

無可追溯性：

修改需求 R1
  ↓ 不知道
哪些測試案例受影響？
  ↓ 只好
全部測試案例都跑（浪費時間）

確認性 vs 迴歸測試比較

執行時機修復後立即每次變更後
