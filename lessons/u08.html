03 8.1 測試技術分類 ⭐⭐⭐

三大測試技術類型 ⭐⭐⭐ 必考

ISTQB 官方分類：

1. 黑箱測試技術（Black-box）

依據：外部文件 關注：功能行為

2. 白箱測試技術（White-box）

依據：內部結構 關注：程式碼、流程

3. 經驗導向測試技術（Experience-based）

依據：測試人員經驗 關注：直覺與知識

考試重點

考題類型1：

以下何者非測試技術類型？ a) 黑箱測試 b) 白箱測試 c) 經驗導向測試 d) 分支測試 ✓（正確答案）

說明：

  • 分支測試是白箱測試技術中的一種方法
  • 不是測試技術的類型

考題類型2：

以下何者為黑箱測試技術？ 答案：等價劃分、邊界值分析、決策表測試、狀態轉換測試

04 8.2 黑箱測試定義 ⭐⭐⭐

核心概念

黑箱測試：

專注受測物的外部可見行為

主要特性

1. 不需要了解內部結構

比喻：使用手機

使用者觀點：
  ✓ 按開機鍵 → 手機開機
  ✓ 點選 App → App 啟動
  ✓ 輸入密碼 → 解鎖成功
  ✗ 不需要知道 CPU 如何運作
  ✗ 不需要知道記憶體如何管理

測試觀點：

  • 只看輸入與輸出
  • 不管內部實作
  • 關注功能正確性

2. 測試案例與結構無關

好處：

系統內部改變
  ↓
演算法優化
  ↓
程式碼重構
  ↓ 但是
測試案例不用改

原因：

  • 測試依據是規格
  • 不是依據程式碼

3. 不要求閱讀程式碼能力

測試人員：

  • 不需要會寫程式
  • 專注功能測試
  • 從使用者角度

優勢：

  • 降低技術門檻
  • 專注業務邏輯

4. 嚴重依賴文件

關鍵：⭐⭐⭐

文件完整性決定測試品質

文件好：

需求明確
  ↓
測試條件清楚
  ↓
測試案例完整
  ↓ 結果
測試品質高

文件差：

需求不清
  ↓
測試條件模糊
  ↓
測試案例不完整
  ↓ 結果
再好的測試技術也沒用

5. 從使用者角度

觀點：

  • 使用者怎麼用
  • 使用者期望什麼
  • 發現使用問題

05 8.3 黑箱測試的依據

測試依據（Test Basis）⭐⭐

定義：

任何可以說明受測系統運作的資訊或文件

包含：

功能需求文件

  • 規格書（Specification）
  • 使用案例（Use Case）
  • 使用者故事（User Story）
  • 功能說明文件

業務規則

  • 商業邏輯
  • 計算公式
  • 處理流程

其他文件

  • 系統架構圖
  • 流程圖
  • 介面規格
  • API 文件

文件完整性的重要性

實例：口罩預約系統

情境：

規格書沒寫：
  「首頁必須在 5 秒內載入」

實際狀況：
  首頁 30 秒才出現
  
問題：
  可以報 Bug 嗎？
  
答案：
  ✗ 不能，因為規格沒寫

結論：

正確答案永遠在規格書

三方關係：

規格書
  ↓ 依據
開發人員 → 實作系統
  ↓ 依據規格
測試人員 → 驗證系統
  
如果規格錯 → 大家都錯

缺乏文件時的做法

當文件不完整或缺乏時：

選項1：要求補充文件

  • 延遲測試
  • 確保品質

選項2：轉用探索性測試

  • 依賴經驗
  • 快速進行
  • 風險較高

06 8.4 黑箱測試的應用範圍

適用測試層級 ⭐

重要原則：

測試類型適用所有測試層級

任何層級都可用黑箱測試：

單元測試層級

測試對象：單一元件
黑箱測試：輸入 → 元件 → 輸出
不看內部實作

整合測試層級

測試對象：元件整合
黑箱測試：驗證介面功能
不看內部溝通方式

系統測試層級

測試對象：完整系統
黑箱測試：端到端功能
不看系統架構

驗收測試層級

測試對象：業務需求
黑箱測試：使用者場景
不看技術實作

適用測試類型

功能性測試 ✓

例子：

  • 登入功能
  • 查詢功能
  • 計算功能
  • 新增/修改/刪除功能

非功能性測試 ✓

例子：

  • 使用性測試
  • 效能測試（使用者觀點）
  • 相容性測試
  • 易用性測試

考題重點：

黑箱測試可用於功能性和非功能性行為測試

07 8.5 黑箱測試技術種類

ISO 29119 標準

總共：12 種黑箱測試技術

Foundation Level 教授：4 種

Foundation Level 四種技術 ⭐⭐⭐ 必考

1. 等價劃分（Equivalence Partitioning）

  • 最基礎、最廣泛
  • 處理值的問題
  • 第4章詳細說明

2. 邊界值分析（Boundary Value Analysis）

  • 延伸自等價劃分
  • 專注邊界測試
  • 第4章詳細說明

3. 決策表測試（Decision Table Testing）

  • 測試複雜邏輯
  • 銀行、保險業常用
  • 第4章詳細說明

4. 狀態轉換測試（State Transition Testing）

  • 測試動態轉換
  • 生命週期測試
  • 第4章詳細說明

進階級額外技術

進階 TA 證照：

  • 延續教上述 4 種（更深入）
  • 額外增加其他技術

ISO 標準的 12 種：

  • 超出 Foundation 範圍
  • 進階證照才教

特殊測試技術

隨機測試（Random Testing）

定義：

按照資料的機率分配進行測試

注意：

  • 不是隨意測試
  • 不是亂測
  • 要依據統計機率分配

例子：

如果系統使用者：
  70% 使用功能 A
  20% 使用功能 B
  10% 使用功能 C

隨機測試：
  測試案例也按 7:2:1 分配

變異測試（Mutation Testing）

新增技術（ISO 29119:2021）

用途：

  • 專門測試 AI 系統
  • AI 測試證照重點

特性：

  • 測試測試案例的品質
  • 確認測試有效性

需求導向測試（Requirements-based Testing）

新增技術（ISO 29119:2021）

用途：

  • 直接從需求產生測試
  • 確保需求覆蓋

08 8.6 覆蓋率概念 ⭐⭐⭐

覆蓋率的本質

定義核心：

已執行項目數 / 總項目數

環島比喻 🚴

問題：如何證明環島成功？

方案1：沿海縣市

台灣沿海縣市：12 個
環島證明：12 個都去過
覆蓋率 = 走過的縣市 / 總縣市數
       = 12 / 12 = 100%

方案2：全部縣市

台灣全部縣市：19 個
走過：17 個
覆蓋率 = 17 / 19 ≈ 89%

方案3：所有鄉鎮

台灣鄉鎮市：364 個
走過：10 個
覆蓋率 = 10 / 364 ≈ 2.7%

方案4：百岳

台灣百岳：100 座
爬過：30 座
覆蓋率 = 30 / 100 = 30%

覆蓋率的關鍵

關鍵1：定義分母 ⭐⭐⭐

問題：

你要覆蓋什麼？

不同測試技術，分母不同：

功能覆蓋率：

分母 = 功能數量
你要先知道：系統有幾個功能？

需求覆蓋率：

分母 = 需求數量
你要先知道：有幾個需求？

等價類別覆蓋率：

分母 = 等價類別數量
你要先知道：分成幾類？

邊界值覆蓋率：

分母 = 邊界值數量
你要先知道：有幾個邊界？

如果分母算錯：

誤以為 15 個縣市
實際是 19 個縣市
  ↓
15/15 = 100%？
  ↓ 錯誤
實際只有 79%（15/19）

關鍵2：覆蓋率 ≠ 通過率 ⭐⭐

覆蓋率：

執行到了

通過率：

執行且通過

例子：四號坑道

情境：
  總長度：10 公里
  走了：8 公里
    ↓ 但是
  中途肚子痛，未完成
    ↓ 結果
  覆蓋率：80%
  通過率：0%

重要：

  • 有執行 ≠ 有通過
  • 覆蓋只看執行
  • 通過要看結果

關鍵3：窮舉測試不可行

環島完美案例：

目標：走遍台灣每個角落
  ↓
台灣面積：36,000 平方公里
  ↓
每平方公里都走到？
  ↓ 不可能
時間、金錢、人力都不足

軟體測試也一樣：

系統所有可能性
  ↓
功能 × 數據 × 環境 × ...
  ↓
組合數 = 天文數字
  ↓ 不可能
窮舉測試

解決方案：

找關鍵項目測試（測試技術的目的）

覆蓋率公式 ⭐⭐⭐

基本公式：

覆蓋率 = 分子 / 分母 × 100%

分子：已執行的項目數
分母：總項目數

每種技術的分母不同：

測試技術分母（覆蓋項目）
等價劃分等價類別數量
邊界值分析邊界值數量
決策表測試決策規則數量
狀態轉換測試狀態數或轉換數
敘述測試可執行敘述行數
分支測試分支數量

重要：

  • 公式簡單：分子除分母
  • 難點：定義分母
  • 先找出分母，才能算覆蓋率

09 8.7 黑箱測試特性詳解

特性1：專注外部行為

觀察重點：

輸入（Input）

  • 使用者輸入
  • 系統接收
  • 參數傳遞

輸出（Output）

  • 系統回應
  • 顯示結果
  • 回傳訊息

行為（Behavior）

  • 功能運作
  • 流程執行
  • 狀態變化

不關心：

  • 程式碼如何寫
  • 演算法如何實作
  • 資料庫如何設計

特性2：基於規格文件

測試依據：

功能規格

需求文件：
  「使用者登入需在 3 秒內完成」
    ↓ 產生測試
  測試案例：
    輸入：帳號、密碼
    期望：3 秒內登入成功

使用案例

UC-001：查詢商品
  1. 輸入商品編號
  2. 按下查詢
  3. 顯示商品資訊
    ↓ 產生測試
  測試每個步驟

使用者故事

As a 使用者
I want to 重設密碼
So that 忘記密碼時可以重新設定
  ↓ 產生測試
驗證重設密碼流程

特性3：測試案例獨立於實作

例子：計算機程式

規格：

輸入：兩個數字 a, b
功能：計算 a + b
輸出：總和

測試案例：

TC1：10 + 20 = 30
TC2：-5 + 3 = -2
TC3：0 + 0 = 0

內部實作改變：

版本1：使用加法運算子
  result = a + b

版本2：使用迴圈累加
  result = a
  for i = 1 to b:
    result = result + 1

版本3：使用遞迴
  add(a, b):
    if b == 0: return a
    return add(a+1, b-1)

測試案例：

完全不用改！
  ↓ 因為
只要輸入輸出一致
內部怎麼做不重要

好處：

  • 系統重構不影響測試
  • 測試案例可重複使用
  • 降低維護成本

10 8.8 黑箱測試的優缺點

優點 👍

1. 不需程式能力

好處：

  • 降低技術門檻
  • 專注業務邏輯
  • 領域專家可參與

2. 從使用者角度

好處：

  • 發現實際使用問題
  • 貼近真實情境
  • 提升使用者滿意度

3. 測試獨立於實作

好處：

  • 程式重構不影響測試
  • 長期可維護
  • 降低維護成本

4. 可早期設計

好處：

  • 規格確定就可設計
  • 不用等程式碼
  • 測試左移

5. 容易理解

好處：

  • 非技術人員也能懂
  • 溝通容易
  • 協作順暢

缺點 👎

1. 無法發現缺失功能

問題：

規格要求：
  會員分級：白金、黃金、銀牌
  優惠：20%、10%、無

程式實作：
  只實作：黃金、銀牌
  遺漏：白金

黑箱測試：
  ✗ 無法發現遺漏
  原因：測試依據規格，不看程式碼
  
白箱測試：
  ✓ 可以發現
  原因：檢視所有程式碼

結論：

功能整個消失，黑箱找不到

2. 嚴重依賴文件品質

文件不完整：

規格缺漏
  ↓
測試案例不完整
  ↓
測試覆蓋不足

文件錯誤：

規格寫錯
  ↓
測試案例跟著錯
  ↓
無法發現真正問題

文件模糊：

規格不清楚
  ↓
測試難以設計
  ↓
測試效果差

3. 無法計算程式碼覆蓋率

限制：

  • 不知道程式碼結構
  • 無法確認程式碼執行
  • 需要白箱測試補充

4. 可能遺漏路徑

問題：

規格描述：功能 A
實際程式：
  - 主要路徑（功能 A）
  - 錯誤處理路徑
  - 例外處理路徑
    ↓
黑箱測試：只測主要路徑
遺漏的路徑：錯誤處理未測

11 8.9 領域知識的重要性 ⭐⭐

為什麼需要領域知識

實例：股票軟體測試

情境：

測試工程師：
  - 剛畢業
  - 從未買過股票
  - 不懂股市運作

任務：
  測試股票交易系統

問題：
  如何測試？
  
困境：
  ✗ 不知道正常流程
  ✗ 不知道異常情況
  ✗ 不知道專業術語
  ✗ 連畫面都看不懂

解決方案：

公司做法：
  送去受訓 3 個月
  學習股票知識
    ↓ 才能
  有效進行測試

領域知識的重要性

1. 理解系統行為

  • 知道功能用途
  • 了解業務邏輯
  • 掌握使用流程

2. 設計有效測試

  • 想到關鍵情境
  • 預期異常狀況
  • 設計完整案例

3. 發現隱藏問題

  • 辨識不合理行為
  • 發現業務邏輯錯誤
  • 提出改善建議

如何累積領域知識

方法1：實際使用

  • 使用類似系統
  • 體驗使用者情境
  • 記錄使用問題

方法2：學習訓練

  • 參加培訓課程
  • 閱讀領域資料
  • 請教領域專家

方法3：觀察分析

  • 觀察使用者行為
  • 分析使用數據
  • 研究競品

12 8.10 黑箱測試的準備

測試前準備

1. 確保文件完整

檢查清單：

✓ 需求是否明確？
✓ 規格是否完整？
✓ 流程是否清楚？
✓ 驗收標準是否定義？

不完整時：

  • 要求補充
  • 進行審查
  • 釐清疑問

2. 理解業務邏輯

準備工作：

✓ 研讀需求文件
✓ 了解使用者角色
✓ 掌握業務流程
✓ 學習領域知識

3. 選擇測試技術

依據情境選擇：

值域範圍：

  • 等價劃分
  • 邊界值分析

複雜邏輯：

  • 決策表測試

狀態轉換：

  • 狀態轉換測試

4. 識別測試條件

從文件中找出：

功能性測試條件：
  - 登入功能
  - 查詢功能
  - 計算功能

非功能性測試條件：
  - 速度
  - 安全性
  - 易用性

13 8.11 黑箱測試與其他技術的關係

與白箱測試互補

黑箱：

  • 檢查「做對的事」
  • 驗證需求實現
  • 使用者角度

白箱：

  • 檢查「把事做對」
  • 驗證程式品質
  • 開發者角度

互補關係：

黑箱測試 + 白箱測試
  ↓
完整的測試覆蓋

與經驗導向測試搭配

黑箱測試：

  • 結構化
  • 系統性
  • 可重複

經驗導向：

  • 靈活性
  • 創造性
  • 補充黑箱不足

搭配使用：

先用黑箱測試：
  - 覆蓋主要功能
  - 系統性測試
    ↓ 再用
經驗導向測試：
  - 發現遺漏
  - 測試特殊情境

14 8.12 實務應用建議

何時使用黑箱測試 ✅

適合情境

1. 文件完整

  • 規格明確
  • 需求清楚
  • 可據以設計測試

2. 功能測試

  • 驗證功能正確性
  • 檢查業務邏輯
  • 確認使用者需求

3. 驗收測試

  • 使用者角度
  • 業務需求驗證
  • 端到端測試

4. 迴歸測試

  • 功能未改變
  • 需要反覆執行
  • 可以自動化

何時不適用黑箱測試 ❌

不適合情境

1. 文件缺乏

沒有規格書
  ↓
無法設計黑箱測試
  ↓ 改用
探索性測試

2. 需要程式碼覆蓋率

要求：80% 分支覆蓋率
  ↓
黑箱測試：無法計算
  ↓ 必須用
白箱測試

3. 內部品質檢查

目標：降低程式複雜度
  ↓
黑箱測試：看不到程式碼
  ↓ 必須用
白箱測試或靜態分析

Q: 以下何者為黑箱測試技術？
A: 等價劃分、邊界值分析、
   決策表測試、狀態轉換測試

題型2：特性判斷

Q: 黑箱測試的特性？
A: 
  ✓ 不需要閱讀程式碼能力
  ✓ 依賴外部文件
  ✓ 從使用者角度
  ✗ 可計算程式碼覆蓋率（錯）

題型3：適用範圍

Q: 黑箱測試可用於哪些層級？
A: 所有測試層級

題型4：覆蓋率計算

Q: 等價類別有 5 個，測試案例覆蓋 3 個，
   覆蓋率是多少？
A: 3/5 = 60%
