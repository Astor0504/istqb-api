01 單元15：使用者故事與驗收標準

02 學習目標

  • 理解使用者故事（User Story）的概念與格式
  • 掌握驗收標準（Acceptance Criteria）的制定
  • 了解INVEST原則

  • 掌握驗收標準的兩種格式
  • 理解驗收測試驅動開發（ATDD）

03 15.1 使用者故事概述 ⭐⭐⭐

什麼是使用者故事

定義 ⚠️：

使用者故事不是需求本身，而是討論需求的媒介或催化劑

核心概念：

傳統需求：
  詳細、完整的需求文件
  
使用者故事：
  只有三句話
  用來引發討論
  
重點：
  ✗ 不等於需求
  ✓ 討論需求的工具
  ✓ 達成共識的媒介

使用者故事格式 ⭐⭐⭐

標準格式（必背）：

As a 【角色】
I want 【功能】
So that 【價值】

作為一個【誰】
我想要【什麼】
以便【為什麼】

範例：

範例1：
As a registered customer
I want to be able to view my previous orders
So that I can keep track of my purchases

作為一個註冊客戶
我想要能夠查看我的歷史訂單
以便我可以追蹤我的購買記錄

────────────────────

範例2：
As an Editor
I want to review content before it is published
So that I can ensure the grammar is correct

作為一個編輯
我想要在內容發布前審查
以便確保語法正確

為什麼這樣寫？ ⭐⭐

As a（角色）：
  → 明確使用者是誰
  
I want（功能）：
  → 需要什麼功能
  
So that（價值）：
  → 為什麼需要（商業價值）
  
好處：
  ✓ 聚焦使用者需求
  ✓ 理解功能價值
  ✓ 避免過度開發

04 15.2 3C模型 ⭐⭐⭐

3C是什麼

Card（卡片）- Conversation（對話）- Confirmation（確認）

Card（卡片）

定義：

在卡片（便利貼）上寫下使用者故事

為什麼用便利貼？

原始理念：
  ✓ 實體卡片
  ✓ 貼在牆上
  ✓ 團隊聚在一起
  ✓ 實體討論
  
現代做法：
  多數改用軟體工具
  (Jira, Azure DevOps等)
  
但精神相同：
  簡潔、可視化

Conversation（對話）⭐⭐⭐

定義：

透過對話討論使用者故事，達成共識

誰參與對話？ ⭐⭐⭐（必考）

必須有3種角色：

1. 商業角色
   └─ 客戶代表、PO（Product Owner）
   
2. 開發角色
   └─ 開發人員
   
3. 測試角色
   └─ 測試人員
   
目的：
  達成共同願景（Shared Vision）

對話目的：

✓ 釐清需求細節
✓ 發現不清楚的地方
✓ 討論實作可行性
✓ 確認測試方法
✓ 達成共識

對話時機：

在撰寫期間：
  不是寫完才討論
  是邊寫邊討論
  
頻率：
  ✓ 頻繁
  ✓ 非正式
  ✓ 輕量級

Confirmation（確認）⭐⭐⭐

定義：

透過驗收標準確認使用者故事完成

核心問題：

如何知道使用者故事「完成」了？
  
答案：
  透過驗收標準（Acceptance Criteria）

測試人員的角色：

在3C中，測試人員最重視：
  
Conversation：
  ✓ 提供測試觀點
  ✓ 識別遺漏細節
  ✓ 發現非功能性需求
  
Confirmation：⭐⭐⭐
  ✓ 制定驗收標準
  ✓ 設計驗收測試
  ✓ 確認完成定義

05 15.3 INVEST原則 ⭐⭐⭐

6個原則（必背）

I - Independent（獨立）：

使用者故事之間應該獨立
  
好處：
  ✓ 可任意排序
  ✓ 可放在不同迭代
  ✓ 不互相依賴
  
範例：
  ✓ 好：登入功能（獨立）
  ✗ 壞：必須先完成註冊才能登入
       （有依賴）

N - Negotiable（可協商）：

使用者故事透過討論產生
不依賴制式合約
  
特點：
  ✓ 可調整
  ✓ 可討論
  ✓ 非固定不變

V - Valuable（有價值）：

每個使用者故事都要有價值
  
問題：
  對誰有價值？
  
答案：
  ✓ 對客戶/使用者有價值
  ✗ 不是對開發者有價值

E - Estimable（可估計）：

可以估計工作量
  
為什麼重要？
  ✓ 才能排優先順序
  ✓ 才能規劃迭代
  ✓ 才能安排資源
  
如何做到？
  → 使用者故事要夠清楚

S - Small（足夠小）：

使用者故事要夠小
  
太大會怎樣？
  ✗ 難以理解本質
  ✗ 難以估計工作量
  ✗ 一次迭代做不完
  
解決：
  拆分成更小的故事

T - Testable（可測試）：⭐⭐⭐

使用者故事必須可測試
  
如何確保？
  ✓ 有明確的驗收標準
  ✓ 可驗證完成與否
  ✓ 結果可觀察

INVEST檢核

檢核表：

檢查使用者故事品質：
  
□ Independent：不依賴其他故事？
□ Negotiable：可討論調整？
□ Valuable：對客戶有價值？
□ Estimable：可估計工作量？
□ Small：夠小可完成？
□ Testable：可測試驗證？

全部✓ → 好的使用者故事
有✗ → 需要改進

06 15.4 驗收標準 ⭐⭐⭐

什麼是驗收標準

定義：

確認使用者故事已完成且可被使用者接受的條件

作用：

回答問題：
  "什麼叫做這個故事做完了？"
  
透過：
  明確的、可測試的條件

何時制定驗收標準 ⭐⭐

時機：

在對話期間（Conversation）
  
不是：
  ✗ 寫完使用者故事才寫
  ✗ 開發完才討論
  
而是：
  ✓ 討論時共同制定
  ✓ 與PO一起決定
  ✓ 團隊達成共識

驗收標準的內容 ⭐⭐⭐

必須包含：

正常情境：
  ✓ 系統應該做什麼
  
異常情境：⭐⭐⭐
  ✓ 系統不應該做什麼
  ✓ 錯誤如何處理
  ✓ 邊界情況
  
兩者都要有！

驗收標準的用途

4大用途 ⭐⭐：

1. 定義範疇

驗收標準越多 = 範疇越大
驗收標準越少 = 範疇越小
  
清楚的範疇：
  ✓ 避免範疇蔓延
  ✓ 控制工作量

2. 達成共識

所有利害關係人：
  ✓ 對完成有共同理解
  ✓ 減少誤解
  ✓ 減少爭議

3. 驗收測試基礎

驗收標準 → 驗收測試案例
  
一對一關係：
  每個驗收標準 = 至少一個測試案例

4. 準確估計

清楚的驗收標準：
  ✓ 更準確估計工作量
  ✓ 更準確估計測試時間

07 15.5 驗收標準的兩種格式 ⭐⭐⭐

格式1：規則導向（Rule-oriented）

定義：

條列式規則說明

特徵：

✓ 使用條列方式
✓ 明確規則
✓ 易於理解

範例：網路銀行即時轉帳

使用者故事：

As a bank customer
I want to transfer money instantly
So that I can send money to friends quickly

驗收標準（規則導向）：

AC1：
若使用者帳戶餘額 ≥ 預轉帳金額
則：
  - 系統應完成該筆轉帳
  - 顯示「轉帳成功」
  - 將交易記錄於使用者的交易明細中
  
AC2：
若使用者帳戶餘額 < 預轉帳金額
則：
  - 系統應顯示「帳戶餘額不足」
  - 拒絕該筆交易

格式2：情境導向（Scenario-oriented）⭐⭐⭐

定義：

使用Given/When/Then格式描述情境

BDD標準格式（必背）：

Given 【前置條件】
When 【觸發事件】
Then 【預期結果】

為什麼用這格式？ ⭐⭐

目的：
  讓測試案例更接近自然語言
  
好處：
  ✓ 使用者看得懂
  ✓ 開發者看得懂
  ✓ 測試者看得懂
  
誰都看得懂！

範例：同樣的轉帳功能

情境1（正常）：

Given 使用者已登入
  And 帳戶餘額為 $1000
When 使用者轉帳 $500
Then 轉帳應該成功
  And 顯示「轉帳成功」訊息
  And 帳戶餘額變為 $500
  And 交易記錄包含此筆轉帳

情境2（異常）：

Given 使用者已登入
  And 帳戶餘額為 $300
When 使用者嘗試轉帳 $500
Then 轉帳應該失敗
  And 顯示「帳戶餘額不足」訊息
  And 帳戶餘額保持 $300
  And 沒有產生交易記錄

變數表技巧 ⭐⭐

使用變數：

Given 使用者已登入
  And 帳戶餘額為 <餘額>
When 使用者轉帳 <金額>
Then <結果>
  And 顯示「<訊息>」
  
Examples:
┌────────┬────────┬──────┬──────────────┐
│ 餘額   │ 金額   │ 結果 │ 訊息         │
├────────┼────────┼──────┼──────────────┤
│ $1000  │ $500   │ 成功 │ 轉帳成功     │
│ $300   │ $500   │ 失敗 │ 帳戶餘額不足 │
│ $1000  │ $0     │ 失敗 │ 金額必須>0   │
│ $1000  │ -$100  │ 失敗 │ 金額必須>0   │
└────────┴────────┴──────┴──────────────┘

好處：

✓ 涵蓋多種情境
✓ 清楚展示邊界
✓ 包含正常與異常
✓ 資料驅動測試

08 15.6 達成共同願景 ⭐⭐⭐

什麼是共同願景（Shared Vision）

定義：

所有利害關係人對使用者故事有一致的理解

為什麼重要？

沒有共同願景：
  開發者理解 ≠ 測試者理解
  測試者理解 ≠ 客戶理解
  
結果：
  ✗ 開發錯誤
  ✗ 測試方向錯
  ✗ 客戶不滿意
  
有共同願景：
  所有人理解一致
  
結果：
  ✓ 照表操課
  ✓ 減少返工
  ✓ 提高品質

如何達成共同願景

敏捷方法：

透過：
  ✓ 頻繁討論
  ✓ 非正式審查
  ✓ 在撰寫期間
  
參與者：
  ✓ 商業角色（客戶代表）⭐
  ✓ 開發角色（開發人員）⭐
  ✓ 測試角色（測試人員）⭐

傳統方法對比：

瀑布式：
  需求寫完 → 正式審查 → 達成共識
  
敏捷：
  撰寫期間 → 持續討論 → 達成共識

討論技巧

腦力激盪（Brainstorming）：

目的：
  激發創意
  發現問題
  
做法：
  ✓ 開放討論
  ✓ 接納所有意見
  ✓ 不批評任何想法

製圖（Modeling）：

視覺化工具：
  ✓ 流程圖
  ✓ 示意圖
  ✓ 原型設計
  
好處：
  更容易理解和討論

09 15.7 驗收測試驅動開發（ATDD）⭐⭐⭐

什麼是ATDD

定義：

從驗收標準開始驅動開發的方法

流程：

Step 1：制定驗收標準
  ↓
Step 2：將驗收標準轉為測試案例
  ↓
Step 3：依據測試案例開發功能
  ↓
Step 4：執行測試，確認通過

ATDD的核心活動

規範研討會（Specification Workshop）：

參與者：
  ✓ 商業代表/PO
  ✓ 開發人員
  ✓ 測試人員
  
目的：
  ✓ 討論使用者故事
  ✓ 制定驗收標準
  ✓ 達成共識
  
產出：
  明確的、可測試的驗收標準

ATDD與BDD的關係

BDD（行為驅動開發）：

關係：
  ATDD著重在驗收測試層級
  BDD著重在系統測試層級
  
共同點：
  ✓ 都用Given/When/Then
  ✓ 都強調自然語言
  ✓ 都促進溝通
  
BDD額外強調：
  軟體的期望行為

自動化驗收測試

目標：

將驗收標準：
  
轉換為：
  可執行的自動化測試
  
好處：
  ✓ 快速回饋
  ✓ 持續驗證
  ✓ 回歸測試

實施：

工具：
  Cucumber、SpecFlow等
  
流程：
  Given/When/Then
    ↓
  自動化測試腳本
    ↓
  執行驗證

10 15.8 Ready與Done定義 ⭐⭐⭐

Definition of Ready（就緒定義）

DOR - Definition of Ready

定義：

使用者故事進入迭代前必須滿足的條件

典型DOR：

□ 使用者故事格式正確（As a/I want/So that）
□ 驗收標準已制定
□ 驗收標準可測試
□ 符合INVEST原則
□ 沒有明顯的依賴問題
□ 工作量已估計
□ 優先級已確定

為什麼需要DOR？

沒有準備好就開始開發：
  ✗ 需求不清楚
  ✗ 做到一半要改
  ✗ 浪費時間
  
有DOR檢查：
  ✓ 確保準備充分
  ✓ 減少返工
  ✓ 提高效率

Definition of Done（完成定義）

DOD - Definition of Done ⭐⭐⭐

定義：

使用者故事完成的標準

典型DOD：

□ 程式碼已撰寫
□ 單元測試已完成且通過
□ 程式碼已審查
□ 功能已整合
□ 所有驗收標準都通過 ⭐⭐⭐
□ 沒有已知的高嚴重度缺陷
□ 文件已更新
□ 可展示給PO

DOD的重要性：

明確定義"完成"：
  ✓ 避免"做完"的爭議
  ✓ 確保品質
  ✓ 可進入下階段

Ready vs Done

流程：

使用者故事
  ↓
檢查DOR（準備好了嗎？）
  ↓ Yes
進入迭代
  ↓
開發與測試
  ↓
檢查DOD（完成了嗎？）
  ↓ Yes
故事完成

11 15.9 實戰案例分析 ⭐⭐⭐

案例1：訂單查詢功能

使用者故事：

As a registered customer
I want to view my previous orders
So that I can keep track of my purchases

驗收標準（規則導向）：

AC1：
使用者登入後點擊「查看訂單歷史」
應顯示所有歷史訂單清單
包含：日期、訂單號、總金額

AC2：
點擊單一訂單
應顯示訂單詳情
包含：商品名稱、價格、數量

AC3：
訂單清單應可依訂單號排序
支援升序與降序

轉換為測試案例：

測試AC1：
Given 客戶已登入帳戶
When 點擊「查看訂單歷史」按鈕
Then 系統顯示所有歷史訂單
  And 每筆訂單包含日期、訂單號、總金額
  
測試AC2：
Given 訂單清單已顯示
When 點擊某筆訂單
Then 系統顯示該訂單詳情
  And 包含所有商品的名稱、價格、數量
  
測試AC3：
Given 訂單清單已顯示
When 點擊「升序排序」
Then 訂單依訂單號由小到大排列

案例2：編輯審查功能

使用者故事：

As an Editor
I want to review content before it is published
So that I can ensure the grammar is correct

驗收標準：

AC1：使用者可用「Editor」角色登入
AC2：編輯可查看現有內容頁面
AC3：編輯可編輯頁面內容
AC4：編輯可新增標記註解
AC5：編輯可儲存變更
AC6：編輯可重新指派給「內容擁有者」角色進行更新

ATDD測試範例：

最佳測試：
Given 以Editor身份登入
When 開啟某內容頁面
  And 編輯頁面內容
  And 儲存變更
Then 變更應該成功儲存
  
為什麼最佳？
  → 涵蓋AC2、AC3、AC5

案例3：電梯門禁系統

使用者故事：

As a Regular or Special user
I want to use my electronic floor card
So that I can access specific floors

驗收標準：

AC1：Regular使用者可進入1-3樓
AC2：4樓僅Special使用者可進入
AC3：Special使用者擁有Regular使用者的所有權限

測試AC3的關鍵：⭐⭐

錯誤理解：
  測試Regular使用者能否進1-3樓 ✗
  （這是測AC1）
  
正確理解：
  測試Special使用者能否進1-3樓 ✓
  （確認Special有Regular的權限）

12 15.10 測試人員在使用者故事中的價值 ⭐⭐⭐

對話階段的貢獻

提供測試觀點：

✓ 如何測試這個功能？
✓ 驗收標準夠明確嗎？
✓ 有沒有遺漏的情境？

識別遺漏細節：

發現問題：
  ✓ 邊界條件未定義
  ✓ 錯誤處理未說明
  ✓ 效能要求未提及

發現非功能性需求：

提醒團隊：
  ✓ 速度要求（3秒內回應）
  ✓ 安全要求（加密傳輸）
