03 5.1 軟體開發生命週期（SDLC）概述

定義

從客戶構想到使用者交付的一系列活動過程

官方定義：

描述軟體開發階段中的相關活動及其邏輯、時間的相關性

SDLC 的三大模型類別 ⭐ 必考

類別英文特性
循序式Sequential線性、階段依序
迭代式Iterative反覆循環、逐步完善
增量式Incremental逐步構建、疊加功能

考試題型：「以下何者『非』SDLC 模型類別？」

記憶提示：循、迭、增（三個即可）

SDLC 對測試的影響

五大影響面向

1. 測試範疇與時機

  • 敏捷：測試從一開始就參與
  • 瀑布：測試在中後期才開始

2. 測試文件詳細度

  • 敏捷：文件最小化（輕量級）
  • 瀑布：以文件為本（詳細完整）

3. 測試技術選擇

  • 敏捷：探索性測試、非正式審查
  • 瀑布：傳統黑箱技術（決策表、狀態轉換）

4. 測試自動化程度

  • 敏捷：強調迴歸測試自動化
  • 瀑布：自動化程度較低

5. 測試人員角色定位

  • 敏捷：需儘早加入、持續回饋
  • 瀑布：在測試階段才深入參與

最佳實務的四個特徵 ⭐

軟體開發流程與測試的最佳實務：

1. 對應性

每個開發活動都有對應的測試活動

例子：

需求階段 → 驗收測試規劃
設計階段 → 系統測試規劃
編碼階段 → 單元測試

2. 獨特性

每個測試層級都有獨特的測試目標
  • 不重複
  • 不遺漏
  • 各有側重

3. 同步性

測試與 SDLC 同步進行
  • 不是等開發完才測試
  • 而是開發到哪，測試到哪

4. 早期性

測試儘早參與
  • 測試左移（Shift Left）
  • 早期發現，成本低

04 5.2 循序式開發模型

特性

  • 線性依序執行階段
  • 沒有重疊
  • 需數月甚至數年才完整交付
  • 動態測試在後期才能執行

瀑布模型（Waterfall Model）⭐

階段流程

需求分析
    ↓
系統設計
    ↓
程式開發
    ↓
系統測試
    ↓
系統維護

核心概念

像瀑布一樣，一階段完成才進入下一階段

比喻：爬樓梯

  • 每階都要踩穩
  • 踩穩才能往上
  • 沒踩穩會跌倒

前提假設（1970年代）

  1. 需求在初期就很穩定
  2. 需求不會頻繁變更
  3. 可以完整掌握需求

優點

  • 結構清楚
  • 易於管理
  • 文件完整

缺點

  1. 需求難以在初期完整掌握
  • 現代需求多變
  • 使用者期望會改變
  • 市場變化快速
  1. 測試在後期才開始
  • 發現問題時修復代價高
  • 錯誤追溯困難
  • 測試時間常被壓縮
  1. 錯誤追溯成本高
測試發現問題
  ↓ 往前追溯
程式碼？
  ↓ 不是
設計？
  ↓ 不是
需求？ ← 終於找到
  ↓
修復成本極高！
  1. 使用者參與度低
  • 只在需求階段參與
  • 後期才看到成果
  • 容易不符期望

V 模型（V-Model）⭐⭐ 重要

模型圖解

需求分析 ←────────→ 驗收測試
    ↓                  ↑
系統設計 ←────────→ 系統測試
    ↓                  ↑
詳細設計 ←────────→ 整合測試
    ↓                  ↑
  編碼   ←────────→ 單元測試

V 的形狀：

  • 左邊：開發活動（下降）
  • 右邊：測試活動（上升）
  • 對應：水平箭頭

核心改善

增加對測試的關注度

改善點：

  1. 每個開發活動都有對應測試
  • 不是只有最後才測試
  • 每階段都規劃對應測試
  1. 提早規劃測試
  • 需求階段就規劃驗收測試
  • 設計階段就規劃系統測試
  1. 測試依據明確
  • 需求 → 驗收測試依據
  • 系統設計 → 系統測試依據
  • 詳細設計 → 整合測試依據
  • 程式碼 → 單元測試依據

驗證 vs 確認（兩個 V）⭐⭐⭐ 必考

驗證（Verification）- 左邊的 V

定義：

檢查工作產物是否符合規範

問題：

「你有沒有照著文件寫的去做？」

例子：

文件寫：1234
  ↓ 驗證
實作：1234 ✓

重點：

  • 只看有沒有照著做
  • 不管內容對不對

測試支援：

  • 靜態測試（審查）
  • 確保文件內容正確

確認（Validation）- 右邊的 V

定義：

檢查工作產物是否符合使用者需求

問題：

「這個東西會不會讓使用者開心？」

例子：

做出系統
  ↓ 確認
使用者滿意 ✓

重點：

  • 實際展示給使用者
  • 確認符合期望

測試支援：

  • 動態測試
  • 實際執行展示

記憶技巧

驗證（Verification）：

  • 文件導向
  • "做對了嗎？"（Do things right）
  • 靜態測試支援

確認（Validation）：

  • 使用者導向
  • "做對的事？"（Do the right things）
  • 動態測試支援

兩者關係

驗證：照文件做 → 確保過程正確
   +
確認：符合期望 → 確保結果有用
   ↓
完整的品質保證

05 5.3 迭代式與增量式模型

與循序式的關鍵差異

循序式（瀑布）

  • 需求穩定
  • 需求為本
  • 功能完整交付

迭代式/增量式

  • 需求多變
  • 價值為本
  • 快速交付核心功能

80/20 法則的啟示 📊

調查發現：

  • 常用功能：20%
  • 很少用功能：80%

新思維：

為何要做 80% 很少用的功能？

解決方案：

  • 優先做 20% 核心功能
  • 快速推出市場
  • 依使用者回饋調整

增量式模型（Incremental）

核心概念

逐步構建，像疊積木

比喻：畫肖像

第一增量：畫鼻子（核心）✓
第二增量：加上眼睛 ✓
第三增量：加上頭部 ✓
第四增量：加上衣服 ✓
第五增量：加上手勢 ✓

特點：

  • 每次增加，前面不改變
  • 明確疊加
  • 進度可見性高

優點

  • 風險分散
  • 容易追蹤進度
  • 核心功能優先

迭代式模型（Iterative）

核心概念

反覆循環，逐步精煉

比喻：畫肖像（不同方法）

第一迭代：粗略輪廓
第二迭代：加細節（整體改變）
第三迭代：加顏色（整體改變）
第四迭代：精緻化（整體改變）
第五迭代：最終完成

特點：

  • 每次迭代會改變
  • 透過回饋逐步完善
  • 最終越來越好

常見模型

  • Prototyping（原型開發）
  • Spiral（螺旋模型）
  • RAD（快速應用開發）

Scrum - 兩者兼具 ⭐

特性：

  • 既有迭代（快速循環）
  • 又有增量（功能疊加）

例子：

Sprint 1：核心功能（增量）+ 調整（迭代）
Sprint 2：新功能（增量）+ 優化（迭代）
Sprint 3：再新功能（增量）+ 改善（迭代）

06 5.4 敏捷開發（Agile）特性

敏捷是一種精神

不是：單一方法論 而是：包含 40+ 種方法論的精神

常見方法論：

  • Scrum ⭐ 最流行
  • XP（Extreme Programming）
  • Kanban
  • FDD（Feature-Driven Development）

敏捷測試特點

1. 工作產物輕量化

  • 不要太多文件
  • 文件維護成本高
  • 強調實際產出

2. 廣泛自動化

  • 迴歸測試自動化
  • 持續整合（CI/CD）
  • 快速回饋

3. 探索性測試

  • 手動探索
  • 發揮測試人員經驗
  • 補充自動化不足

07 5.5 測試驅動開發（TDD、BDD、ATDD）

TDD（Test-Driven Development）⭐

核心概念

先寫測試，再寫程式碼

流程：

1. 寫測試案例（會失敗）
2. 寫最少的程式碼讓測試通過
3. 重構程式碼
4. 重複

優點

  • 釐清需求
  • 避免開發人員偷懶不寫測試
  • 持續確保品質

缺點

  • 開發速度較慢
  • 需要紀律
  • 測試程式碼也需維護

主要用途

  • 單元測試層級

BDD（Behavior-Driven Development）

核心概念

用自然語言撰寫測試

Given-When-Then 格式：

Given（前提）：使用者已登入
When（動作）：點擊「查詢」按鈕
Then（結果）：顯示查詢結果頁面

優點

  • 可讀性高
  • 非技術人員也看得懂
  • 促進溝通

主要用途

  • 系統測試層級

ATDD（Acceptance Test-Driven Development）

核心概念

先定義驗收標準，再開發

流程：

1. 與客戶討論驗收標準
2. 將驗收標準轉為測試案例
3. 開發功能
4. 執行驗收測試

優點

  • 明確驗收標準
  • 減少誤解
  • 確保符合期望

主要用途

  • 驗收測試層級

08 5.6 DevOps 與測試

DevOps 核心概念

定義

開發（Dev）+ 運營（Ops）的協作文化

核心要素

1. 文化轉變

  • 打破開發、測試、運營的孤島
  • 相互尊重、同等價值
  • 共同對品質負責

2. 快速回應

  • 快速發布
  • 快速回饋
  • 快速修復

3. 工具整合

  • CI/CD 流水線
  • 版本控制
  • 自動化測試

4. 測試自動化

  • 迴歸測試自動化
  • 持續測試
  • 快速回饋

DevOps 對測試的好處 ⭐ 需背

必背的 5 個好處：

  1. 快速回饋程式碼品質
  • 提交程式碼立即測試
  • 立即知道有無問題
  1. 促進測試工作的可見性
  • 測試結果透明化
  • Dashboard 即時顯示
  1. CI/CD 自動化流程
  • 測試環境更一致
  • 減少人為錯誤
  1. 更多時間專注非功能性測試
  • 自動化處理重複工作
  • 人力專注創造性測試
  1. 減少重複手動測試
  • 降低迴歸風險
  • 提高效率

DevOps 的風險與挑戰 ⭐ 需背

必背的風險：

  1. 定義與建立交付管道
  • 需要時間與專業
  • 初期投資大
  1. 導入 CI/CD 工具
  • 工具學習曲線
  • 整合現有流程
  1. 建立測試自動化
  • 需要額外資源
  • 持續維護成本

高投資報酬率的自動化 💰

ISTQB 建議優先自動化：

1. 迴歸測試

  • 重複執行頻率高
  • 自動化效益最大

2. 冒煙測試（Smoke Testing）

定義：核心功能測試

為何優先：

  • 核心功能最重要
  • 確保基本可用
  • 快速驗證

09 5.7 測試左移（Shift Left）

核心概念

測試越早介入越好

原理

成本比較（研究數據）

2002 年調查：

需求階段發現：修復成本 = 1
發布階段發現：修復成本 = 30
差距：30 倍

2010 年調查：

需求階段發現：修復成本 = 1
發布階段發現：修復成本 = 100
差距：100 倍

2025 年預估：

差距可能超過 200 倍
（因為系統更複雜）

視覺化理解

    成本
     ↑
 200│              ●發布階段
 150│          ●
 100│      ●
  50│  ●
   1│●─────────────→ 時間
    需求 設計 編碼 測試 發布

測試左移的做法

1. 需求審查

  • 測試人員參與需求討論
  • 審查需求文件
  • 提出可測試性建議

2. 持續整合

  • 程式碼提交前自動測試
  • 快速發現問題

3. 靜態測試

  • 動態測試前先做靜態測試
  • 程式碼審查
  • 靜態分析工具

4. 提早非功能性測試

  • 不等到系統測試才做
  • 單元測試階段就開始

挑戰

1. 組織文化

障礙：

  • 傳統觀念：測試在後期
  • 需要說服管理層

解決：

  • 展示成本節省數據
  • 分享成功案例

2. 技能提升

需求：

  • 測試人員需學習靜態測試
  • 需要審查技能

3. 資源投入

前期：

  • 時間增加
  • 成本上升

長期：

  • 總成本下降
  • 品質提升

10 5.8 測試層級（Test Levels）⭐⭐⭐

定義

一群有組織且可管理的測試活動

傳統：4 個測試層級（3.1 版）

舊版：

  1. 單元測試
  2. 整合測試
  3. 系統測試
  4. 驗收測試

現行：5 個測試層級（4.0 版）⭐ 必考

新版（考試以此為準）：

  1. 單元測試（Component Testing）
  2. 元件整合測試（Component Integration Testing）
  3. 系統測試（System Testing）
  4. 系統整合測試（System Integration Testing）
  5. 驗收測試（Acceptance Testing）

改變：整合測試分為兩個

樂高積木比喻 🧱

單元測試：      一個積木是否正常？
                 [🧱] ✓

元件整合測試：  兩個積木疊起來會倒嗎？
                 [🧱]
                 [🧱] ✓

系統測試：      整個作品完整嗎？
                 [🧱][🧱]
                 [🧱][🧱]
                 [🧱][🧱] ✓

系統整合測試：  與外部作品結合呢？
                 你的作品 + 別人的作品 ✓

驗收測試：      使用者滿意嗎？
                 客戶驗收 ✓

重要原則 ⭐

每個測試層級都可執行各種測試類型

錯誤觀念：

  • ❌ 效能測試只能在系統測試做
  • ❌ 安全測試只能在系統測試做

正確觀念：

  • ✓ 單元測試也可以測效能
  • ✓ 整合測試也可以測安全
  • ✓ 各層級都可執行功能性/非功能性測試

11 5.9 單元測試（Component Testing）

定義

確保受測對象獨立運作正常

核心概念：隔離測試

例子：

元件 A、元件 B 要整合
  ↓ 先確認
元件 A 單獨測試 ✓
元件 B 單獨測試 ✓
  ↓ 才能
A + B 整合測試

測試重點

1. 功能性

  • 邏輯正確
  • 計算正確
  • 資料處理正確

2. 資料品質

  • 資料結構正確
  • 變數宣告正確
  • 初始化正確

3. 程式碼品質

  • 可讀性
  • 可維護性
  • 符合編碼標準

責任歸屬

誰負責：開發人員 ⭐

原因：

  • 對程式碼最熟悉
  • 可立即修復
  • 成本最低

缺陷處理

標準做法

發現 Bug → 立即修復

不進入 Bug 管理系統

原因：

  • 快速修復
  • 避免流程延遲

例外：可考慮進入 Bug 系統

好處：

  • 累積數據
  • 分析常犯錯誤
  • 根本原因分析
  • 流程改善

壞處：

  • 增加行政作業
  • 降低修復速度

測試技術

主要使用

  • 白箱測試技術
  • 敘述測試（Statement Testing）
  • 分支測試（Branch Testing）

必要工具

  • 單元測試框架
  • JUnit（Java）
  • NUnit（.NET）
  • pytest（Python）
  • Jest（JavaScript）

測試驅動開發（TDD）在單元測試

TDD 流程

1. 寫測試案例（紅燈）
2. 寫程式碼（綠燈）
3. 重構
4. 重複

優點

  • 測試案例先行
  • 幫助釐清需求
  • 確保可測試性
  • 避免開發人員不寫測試

缺點

  • 系統本質未定義就要寫測試
  • 需要紀律
  • 初期較慢

12 5.10 整合測試（Integration Testing）

為何分成兩種？（4.0 新版）

元件整合測試（Component Integration Testing）

測試對象：系統內部元件之間的整合

例子：

使用者介面（UI）
    ↕ 整合
業務邏輯層（Business Logic）
    ↕ 整合
資料存取層（Data Access）

系統整合測試（System Integration Testing）

測試對象：系統與外部系統的整合

例子：

你的系統
    ↕ 整合
第三方金流（PayPal）
    ↕ 整合
物流系統（宅配通）

責任歸屬 ⭐ 重要

測試層級負責人
單元測試開發人員
元件整合測試開發人員
系統測試測試人員
系統整合測試測試人員
驗收測試使用者

記憶技巧：

  • 元件 → 內部 → 開發人員熟悉 → 開發負責
  • 系統 → 外部 → 需獨立觀點 → 測試負責

整合策略

1. 大爆炸整合（Big Bang）❌ 不建議

做法：所有元件同時整合

例子：

A + B + C + D → 一起整合

缺點：

  • 問題發生，不知道哪裡錯
  • 難以定位
  • 風險極高

案例：南山人壽神蹟專案

  • 40 個系統同時整合
  • 上線後問題百出
  • 無法回退舊系統
  • 被罰款、董事長下台

結論：除非系統極小，否則不用

2. 逐步整合（Incremental Integration）✓ 建議

核心概念：

一次只整合少數元件
由下而上（Bottom-Up）
步驟1：A + B
步驟2：A+B 的結果 + C
步驟3：A+B+C 的結果 + D
由上而下（Top-Down）
步驟1：主程式 + A
步驟2：主+A 的結果 + B
步驟3：主+A+B 的結果 + C

優點：

  • 問題範圍小
  • 容易定位
  • 風險可控

例子：

情境：3 個功能整合出問題

大爆炸：A+B+C 一起 → 不知道誰的問題
逐步整合：
  A+B → 沒問題 ✓
  A+B+C → 有問題 ✗
  結論：問題在 C

13 5.11 系統測試（System Testing）

定義

測試完整系統的行為與能力

測試重點

1. 功能性測試

  • 完整的業務流程
  • 端到端（End-to-End）測試

例子：購物流程

選商品 → 加入購物車 → 結帳 → 付款 → 確認

2. 非功能性測試

  • 效能效率
  • 安全性
  • 易用性
  • 相容性
  • 可靠性

執行環境

理想：

  • 代表性測試環境
  • 盡量接近生產環境

可接受：

  • 透過模擬器
  • 虛擬化技術
  • 服務虛擬化（Service Virtualization）

責任歸屬

誰負責：獨立測試團隊 ⭐

原因：

  • 需要客觀獨立性
  • 避免開發者盲點
  • 多元視角

測試依據

  • 系統規格
  • 軟體規格
  • 風險分析報告
  • 使用案例
  • 使用者故事
