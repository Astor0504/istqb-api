03 12.1 狀態轉換測試概述 ⭐⭐⭐

什麼是狀態轉換測試

定義：

用於測試具有動態轉換特性的系統 模擬現實世界中狀態變化的行為

適用場景：

✓ 交通號誌（紅→黃→綠）
✓ 會員等級（銅→銀→金→白金）
✓ 訂單狀態（待付款→已付款→已出貨→已完成）
✓ 電器開關（開→關→開）
✓ 遊戲角色狀態（正常→中毒→治療→正常）
✓ 帳戶狀態（正常→凍結→解凍）

為什麼需要狀態轉換測試

現實世界的動態性：

靜態系統：
  輸入 → 處理 → 輸出
  （一次性）
  
動態系統：
  狀態A → 事件 → 狀態B
  狀態B → 事件 → 狀態C
  （持續變化）

測試重點：

✓ 每個狀態都正確
✓ 轉換關係正確
✓ 不該轉換的不會轉
✓ 順序邏輯正確

04 12.2 狀態轉換的核心元素 ⭐⭐⭐

五大核心元素 ⭐⭐⭐ 必記

1. 狀態（State）

定義：

系統在某時間點的特定條件或情況

圖形表示：

  • 矩形
  • 橢圓形
  • 圓角矩形

範例：

會員系統：
  S1: 銅級會員
  S2: 銀級會員
  S3: 金級會員
  S4: 白金會員
  
訂單系統：
  S1: 待付款
  S2: 已付款
  S3: 已出貨
  S4: 已完成

2. 轉換（Transition）

定義：

從一個狀態變化到另一個狀態

圖形表示：

  • 箭頭（→）
  • 有方向性 ⭐⭐⭐

重要觀念：

單向箭頭：
  A → B（只能從A到B，不能從B回A）
  
雙向箭頭：
  A ⇄ B（可以A→B，也可以B→A）
  
可能的畫法：
  A → B（分開畫）
  或
  A ← B
  或
  A ⇄ B（一個箭頭表示兩個轉換）⭐

3. 事件（Event）

定義：

觸發狀態轉換的動作或條件

位置：

寫在箭頭上方的文字

範例：

S1 --[付款]--> S2
  ↑    ↑
狀態  事件  狀態

關係：

沒有事件 → 不會轉換
  
S1不會憑空變成S2
  必須透過某事件觸發

4. 約束條件（Guard Condition）

定義：

事件發生時的額外限制或條件

表示方式：

寫在方括號 [ ] 內

範例：

不及格 --[考試[≥65分]]--> 及格
       ↑         ↑
      事件    約束條件
      
意思：
  觸發事件：考試
  但要符合：分數≥65分
  才能轉換到及格狀態

為什麼需要約束條件？

沒有約束：
  考試 → 及格（只要考就會過？）
  
有約束：
  考試[≥65分] → 及格（合理！）

5. 結果（Action/Result）

定義：

狀態轉換時產生的輸出或動作

範例：

銅級 --[消費滿10次/送T恤]--> 銀級
                    ↑
                   結果

05 12.3 狀態圖製作方法 ⭐⭐⭐

基本符號

起始狀態：

● （實心小圓圈）

結束狀態：

◉ （實心圓圈外包空心圓圈）

一般狀態：

┌─────────┐
│ 狀態名稱 │
└─────────┘

轉換：

S1 --[事件[條件]/結果]--> S2

狀態圖範例

範例1：及格/不及格系統

簡單狀態圖：

     ●
     ↓
┌─────────┐
│ 不及格   │
└─────────┘
     │[考試[≥65分]]
     ↓
┌─────────┐
│  及格    │
└─────────┘
     ↓
     ◉

解讀：

1. 系統從不及格狀態開始
2. 當考試且分數≥65分
3. 轉換到及格狀態
4. 結束

範例2：會員升降級

複雜狀態圖：

     ●
     ↓
┌─────────┐
│  銅級    │←───────┐
└─────────┘        │
     │[消費20次]   │[降級]
     ↓             │
┌─────────┐        │
│  銀級    │────────┤
└─────────┘        │
     │[消費30次]   │[降級]
     ↓             │
┌─────────┐        │
│  金級    │────────┘
└─────────┘
     ↓
     ◉

重點觀察：

✓ 可向上升級
✓ 可向下降級
✓ 雙向轉換
✓ 不同條件觸發

範例3：自我轉換

狀態圖：

┌─────────┐
│ 登入狀態 │ ⤾ [重新整理]
└─────────┘

意義：

自我轉換：
  從同一狀態回到同一狀態
  
範例：
  已登入 → 重新整理 → 仍在登入狀態

06 12.4 狀態表製作方法 ⭐⭐⭐

什麼是狀態表

定義：

用表格方式呈現狀態轉換關係

優點：

相較狀態圖：
  ✓ 容易找出無效轉換
  ✓ 資訊更完整
  ✓ 容易檢查完整性

狀態表結構

基本格式：

        事件A  事件B  事件C
狀態1     S2     -      S3
狀態2     S1     S3     -
狀態3     -      S1     END
END       -      -      -

說明：
  橫軸：當前狀態
  縱軸：發生的事件
  內容：下一個狀態（-表示無效）

狀態圖與狀態表對應

狀態圖：

S1 --[事件A]--> S2

對應狀態表：

      事件A
S1     S2
  ↑     ↑
當前  下一個
狀態  狀態

完整範例：客房訂房系統

狀態圖：

  ● → S1:申請中
       ↓ [有空房]     ↓ [無空房]
    S2:已確認      S3:候補中
       ↓ [付款]     ↓ [有空房]
       END    ←───────┘

狀態表：

              有空房  無空房  付款  取消
S1:申請中      S2     S3     -     -
S2:已確認      -      -     END   END
S3:候補中      S2     -      -    END
END           -      -      -     -

表格解讀：

S1狀態：
  有空房 → S2（確認）
  無空房 → S3（候補）
  付款   → -（無效）
  取消   → -（無效）

07 12.5 狀態轉換測試的覆蓋率 ⭐⭐⭐

三種覆蓋率 ⭐⭐⭐ 必記

覆蓋率1：狀態覆蓋

定義：

測試案例執行到的狀態數 / 總狀態數

公式：

狀態覆蓋率 = 經過的狀態數 / 總狀態數 × 100%

範例：

狀態圖有4個狀態：S1, S2, S3, S4

測試：S1 → S2 → S3 → S4
  
覆蓋：4個狀態
覆蓋率：4/4 = 100%

最小測試數：

通常1個測試案例就能達到100%狀態覆蓋
  
方法：
  找一條路徑經過所有狀態

缺點 ⚠️：

只關注狀態有沒有到達
不關注箭頭（轉換）
  
可能遺漏：
  某些轉換關係未測試

覆蓋率2：有效轉換覆蓋

定義：

測試的有效轉換數 / 總有效轉換數

什麼是有效轉換？

有箭頭的轉換 = 有效轉換
沒箭頭的轉換 = 無效轉換

公式：

有效轉換覆蓋率 = 測試的箭頭數 / 總箭頭數 × 100%

範例：

狀態圖：
  S1 → S2 → S3 → S4
       ↓         ↑
       └─────────┘
       
箭頭數（有效轉換）= 5
  1. S1→S2
  2. S2→S3
  3. S3→S4
  4. S2→S2（自我轉換）
  5. S3→S2

測試設計：

需要幾個測試案例才能100%覆蓋？
  
答案：看能否一條路徑走完所有箭頭
  
可能需要：2-3個測試案例

覆蓋率3：所有轉換覆蓋

定義：

(有效轉換 + 無效轉換) 覆蓋率

公式：

所有轉換覆蓋率 = 測試的轉換數 / 所有轉換數 × 100%

所有轉換數 = 有效轉換 + 無效轉換

無效轉換：

定義：
  沒有箭頭的轉換
  系統不應該發生的轉換
  
為什麼要測？
  確保系統不會錯誤轉換

範例：

狀態圖：
  銅級 → 銀級 → 金級
  
有效轉換：
  銅→銀 ✓
  銀→金 ✓
  
無效轉換（要測！）：
  金→銀 ✗
  金→銅 ✗
  銀→銅 ✗
  
目的：
  確保不會意外降級

找無效轉換的方法：

用狀態表更容易！
  
狀態表中的 "-" = 無效轉換
  
     事件A  事件B
S1    S2     -    ← "-"就是無效
S2    -      S3

計算範例：

4個狀態，3個事件
  
理論總轉換 = 4 × 3 = 12
有效轉換 = 5（有箭頭）
無效轉換 = 12 - 5 = 7
  
要100%所有轉換覆蓋：
  需測試 5+7 = 12 個轉換

08 12.6 N-Switch 覆蓋 ⭐⭐⭐

N-Switch 概念 ⭐⭐⭐ 進階

定義：

N 代表兩個狀態之間夾雜的狀態數量

0-Switch（就是有效轉換）⭐⭐⭐

概念：

兩個狀態之間：
  沒有夾雜其他狀態
  
S1 → S2
  ↑
直接轉換，N=0

特徵：

0-Switch = 有效轉換
  
計算：
  箭頭數量 = 0-Switch 數量

範例：

狀態圖：
  S1 → S2 → S3
       ↓     ↑
       └─────┘
       
0-Switch（箭頭）：
  1. S1→S2 ✓
  2. S2→S3 ✓
  3. S3→S2 ✓
  
總共3個

1-Switch

概念：

兩個狀態之間：
  夾雜1個其他狀態
  
S1 → S2 → S3
├──────────┤
    N=1

範例：

S1 → S2 → S3

1-Switch：
  S1→S2→S3（S1到S3中間經過S2）
  
不是1-Switch：
  S1→S2（只經過1個轉換，N=0）
  S2→S3（只經過1個轉換，N=0）

2-Switch

概念：

兩個狀態之間：
  夾雜2個其他狀態
  
S1 → S2 → S3 → S4
├───────────────┤
       N=2

N-Switch 的重要性

為什麼要測N-Switch？

Bug可能在：
  ✗ 直接轉換時出現
  ✓ 轉換路徑組合時出現
  
範例：
  A → B：正常 ✓
  A → C → B：異常 ✗
    ↑
  特定路徑才觸發的Bug

考試範圍 ⚠️：

Foundation考試：
  最多考到 1-Switch
  
不會考：
  2-Switch, 3-Switch...

09 12.7 狀態轉換測試案例設計 ⭐⭐⭐

設計方法：路徑追蹤

方法1：追求最少測試數

目標：

用最少測試案例達到目標覆蓋率

技巧：

找一條路徑：
  盡可能經過最多箭頭
  
如果一條不夠：
  找第二條補足未覆蓋的

範例：

狀態圖：
  S0 → S1 → S2 → S3 → S4
       ↓         ↑
       └─────────┘
       
有效轉換：
  1. S0→S1
  2. S1→S2
  3. S2→S3
  4. S3→S4
  5. S2→S2（自我）
  6. S3→S2
  
設計：
  TC1: S0→S1→S2→S3→S4
    ↑ 覆蓋：1,2,3,4
    
  TC2: S0→S1→S2→S2→S3→S2→...
    ↑ 覆蓋：5,6
    
  最少2個測試案例

方法2：系統化枚舉

步驟：

1. 列出所有有效轉換
2. 為每個設計測試路徑
3. 合併相同部分

測試案例表示方式

方式1：事件序列

TC1: 申請 → 有空房 → 付款

方式2：狀態序列

TC1: S1 → S2 → END

方式3：詳細描述

TC1:
  初始狀態：申請中
  步驟1：觸發"有空房"事件 → 確認狀態
  步驟2：觸發"付款"事件 → 結束

10 12.8 覆蓋率計算實例 ⭐⭐⭐

實例1：狀態覆蓋

題目：

狀態圖有3個狀態：S1, S2, S3
問：最少需要幾個測試案例達到100%？

解答：

答案：1個

路徑：S1 → S2 → S3
  ↑ 經過所有狀態

覆蓋率：3/3 = 100% ✓

實例2：有效轉換覆蓋

題目：

狀態圖：
  S0 → S1 → S2
       ↓
       S3
       
有幾個有效轉換？
需要幾個測試案例？

解答：

箭頭數（有效轉換）：
  1. S0→S1
  2. S1→S2
  3. S1→S3
  總共：3個
  
最少測試數：
  TC1: S0→S1→S2（覆蓋1,2）
  TC2: S0→S1→S3（覆蓋3）
    ↑ 但1重複
    
  實際：2個測試案例

實例3：所有轉換覆蓋

題目：

3個狀態，2個事件
  
狀態表：
      事件A  事件B
S1     S2     -
S2     -      S3
S3     S1     -

解答：

Step 1：數有效轉換
  S1→(A)→S2
  S2→(B)→S3
  S3→(A)→S1
  有效：3個
  
Step 2：數無效轉換
  S1→(B)→?（無）
  S2→(A)→?（無）
  S3→(B)→?（無）
  無效：3個
  
總轉換數：3+3=6
  
要測試：
  ✓ 3個有效（確認能轉）
  ✓ 3個無效（確認不會轉）

11 12.9 實戰案例分析

案例1：電商訂單狀態

需求：

訂單狀態轉換：
  建立 → 付款 → 出貨 → 完成
       ↓               ↓
      取消            退貨

狀態表：

         付款  取消  出貨  退貨  完成
建立      付款  取消  -     -     -
付款       -    取消  出貨  -     -
出貨       -     -    -    退貨  完成
取消       -     -    -     -     -
退貨       -     -    -     -     -
完成       -     -    -     -     -

測試設計：

正常流程：
  TC1: 建立→付款→出貨→完成
  TC2: 建立→取消
  TC3: 付款→取消
  TC4: 出貨→退貨
  
異常測試（無效轉換）：
  TC5: 建立狀態能否直接出貨？（應失敗）
  TC6: 完成後能否取消？（應失敗）

案例2：ATM狀態

狀態圖：

  待機 → 插卡 → 驗證PIN
         ↓       ↓[正確]  ↓[錯誤3次]
        退卡    交易     鎖卡
                 ↓
                退卡

重點測試：

✓ PIN輸入錯誤1次 → 繼續
✓ PIN輸入錯誤2次 → 繼續
✓ PIN輸入錯誤3次 → 鎖卡 ⭐
✓ 正確PIN → 進入交易
✓ 交易完成 → 退卡
✓ 中途取消 → 退卡

案例3：會員等級轉換

現代電商特性：

以前：
  只能升級，不能降級
  銅 → 銀 → 金 → 白金（單向）
  
現在：
  可升可降（雙向）
  銅 ⇄ 銀 ⇄ 金 ⇄ 白金
  
原因：
  鼓勵持續消費

測試重點：

升級條件：
  消費滿額 → 升級
  
降級條件：
  N天未消費 → 降級
  
測試：
  ✓ 達標能升
  ✓ 未達標不升
  ✓ 超時會降
  ✓ 持續消費不降

12 12.10 考試題型與解題技巧 ⭐⭐⭐

題型1：數箭頭（K1）

標準題型：

Q: 此狀態圖有幾個有效轉換？

解法：

數箭頭數量即可
  
注意：
  ⇄ 雙向箭頭 = 2個轉換
  → 單向箭頭 = 1個轉換

題型2：最少測試數（K3）

題目：

給定狀態圖，問：
  達到100%有效轉換覆蓋
  最少需要幾個測試案例？

解題步驟：

Step 1：數箭頭（如7個）

Step 2：找路徑
  嘗試一條路徑走完所有箭頭
  
Step 3：補足
  第一條走不完的
  用第二條補
  
Step 4：計算
  需要的測試數

技巧 ⭐⭐⭐：

觀察：
  ✓ 有沒有自我轉換（要特別測）
  ✓ 有沒有雙向（要來回測）
  ✓ 有沒有分支（要分開測）

題型3：判斷可行路徑（K3）

題目範例：

給四個事件序列，問哪個可行？

選項A: Add, Add, Add, Remove
選項B: Add, Add, Add, Add
選項C: Remove, Add, Add
選項D: Add, Remove, Remove

解法：

錯誤1：忽略箭頭方向性

錯誤觀念：

看到：A ──── B
  
誤以為：
  可以 A→B
  也可以 B→A

正確觀念 ⭐⭐⭐：

箭頭有方向性！
  
A → B：只能 A到B
A ← B：只能 B到A  
A ⇄ B：雙向都可
  
一個雙向箭頭 = 2個轉換

錯誤觀念：

100%狀態覆蓋 
  = 100%轉換覆蓋 ✗

✅ 本課任務
