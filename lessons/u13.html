03 13.1 白箱測試概述 ⭐⭐⭐

什麼是白箱測試

定義：

基於系統內部結構設計測試的技術 關注程式碼、流程、架構等內部構造

別名：

✓ 白箱測試（White-box Testing）
✓ 結構化測試（Structural Testing）
✓ 透明盒測試（Clear-box Testing）

黑箱 vs 白箱 ⭐⭐⭐

比較表：

特性          黑箱測試            白箱測試
────────────────────────────────────
依據          外部規格            內部結構
關注點        做什麼（What）      如何做（How）
需要看程式碼  不需要              需要
測試者        測試人員            開發/測試人員
時機          任何階段            有程式碼後
文件依賴      高度依賴            不依賴
自動化        部分可              容易自動化
覆蓋率        可計算              可計算

資源劃分圖：

           規格/文件
               ↓
         【黑箱測試】
         
           程式碼
               ↓
         【白箱測試】
         
          知識/經驗
               ↓
        【經驗導向測試】

04 13.2 白箱測試的核心概念

結構是什麼？

在不同測試層級：

單元測試層級：⭐⭐⭐

結構 = 程式碼內部
  ✓ 每一行敘述（Statement）
  ✓ 分支（Branch）
  ✓ 決策（Decision）
  ✓ 路徑（Path）

整合測試層級：

結構 = 介面/API
  ✓ 函數呼叫
  ✓ 介面互動
  ✓ 資料流

可執行敘述 vs 不可執行

可執行敘述 ⭐⭐⭐：

1  int x = 5;           ✓ 可執行
2  y = x + 10;          ✓ 可執行
3  if (x > 0)           ✓ 可執行
4    print("正數");     ✓ 可執行
5                       ✗ 空白行
6  // 這是註解         ✗ 註解
7  {                    ✗ 括號
8  }                    ✗ 括號

判斷原則：

可執行：
  ✓ 宣告敘述
  ✓ 賦值敘述
  ✓ 控制敘述（if, for, while）
  ✓ 函數呼叫
  ✓ 運算敘述
  
不可執行：
  ✗ 空白行
  ✗ 註解
  ✗ 單獨的括號

考試注意 ⚠️：

有標行號的 → 要計算
沒標行號的 → 不計算
  
若都沒標行號：
  → 全部都要算

05 13.3 敘述測試（Statement Testing）⭐⭐⭐

定義與目標

定義：

設計測試案例執行程式碼中的每一行可執行敘述

目標：

100% 敘述覆蓋率 =
  每一行可執行敘述都被執行至少一次

前提假設：

如果程式碼有Bug：
  → Bug一定存在於某行程式碼中
  
如果每行都執行到：
  → Bug應該會被觸發
  
（但這假設有缺陷！後面會說明）

覆蓋率計算 ⭐⭐⭐

公式：

敘述覆蓋率 = 執行到的敘述數 / 總可執行敘述數 × 100%

範例1：簡單程式

1  read(x)           ← 會執行
2  read(y)           ← 會執行
3  if (x > y)        ← 會執行
4    z = x           ← 看輸入值
5  else
6    z = y           ← 看輸入值
7  print(z)          ← 會執行

測試案例：

TC1: x=10, y=5
  執行：1,2,3,4,7
  覆蓋：5/5 = 100% ✓
  
為什麼100%？
  第6行（z=y）沒執行，但它與第4行（z=x）
  只會執行其中一個，所以只需執行5行

計算要點：

總敘述數 = 5（不是7）
  因為：
    行3的if算1行
    行4和行6只需執行一個
    
實際可執行敘述：
  read(x), read(y), if判斷, z賦值, print(z)
  共5行

範例2：100%覆蓋率實例

1  input(A)
2  if (A > 10)
3    B = 20
4  else
5    B = 5
6  print(B)

測試：

TC: A = 15
  
執行路徑：
  1 → 2 → 3 → 6
  
覆蓋率：
  執行：1,2,3,6（4行）
  總數：4行（2,3,5,6只需執行其一）
  覆蓋率：4/4 = 100% ✓

06 13.4 分支測試（Branch Testing）⭐⭐⭐

定義與目標

定義：

設計測試案例執行程式碼中的每一個分支

什麼是分支：

分支 = 控制的轉移
  
兩種分支：
  1. 無條件分支
  2. 有條件分支

無條件分支

定義：

執行A一定會執行B

範例：

1  int x;      ← 無條件
2  int y;      ← 一定執行
3  x = 5;      ← 一定執行

特徵：

✓ 循序執行
✓ 沒有判斷
✓ 必然發生

有條件分支

定義：

根據條件決定執行哪個分支

範例：

if (x > 0)
  A          ← 條件分支1
else
  B          ← 條件分支2
  
執行A或B取決於條件

常見結構：

✓ if-else
✓ switch-case
✓ 迴圈（while, for）
✓ 三元運算子（? :）

分支覆蓋率計算 ⭐⭐⭐

公式：

分支覆蓋率 = 執行到的分支數 / 總分支數 × 100%

範例1：基本if結構

1  read(x)
2  read(y)
3  if (x > y)
4    z = 0
5  print(z)

分支分析：

分支1: 3→4（x>y為真）
分支2: 3→5（x>y為假）
總分支數：2

測試：

TC1: x=10, y=5
  路徑：1→2→3→4→5
  覆蓋：分支1
  覆蓋率：1/2 = 50%
  
TC2: x=3, y=8
  路徑：1→2→3→5
  覆蓋：分支2
  覆蓋率：1/2 = 50%
  
TC1+TC2:
  覆蓋率：2/2 = 100% ✓

範例2：複雜if結構

1  read(x)
2  read(y)
3  if (x > y)
4    if (y > 0)
5      z = 1
6  print(z)

分支分析：

第一個if（行3）：
  分支1: 3→4（x>y真）
  分支2: 3→6（x>y假）
  
第二個if（行4）：
  分支3: 4→5（y>0真）
  分支4: 4→6（y>0假）
  
總分支數：4

07 13.5 敘述覆蓋 vs 分支覆蓋 ⭐⭐⭐

包含關係 ⭐⭐⭐ 必記

重要定理：

100% 分支覆蓋 
  → 必然包含 100% 敘述覆蓋

反之不成立！

100% 敘述覆蓋 
  → 不保證 100% 分支覆蓋

為什麼分支更強？

範例說明：

1  read(x)
2  if (x == 0)
3    y = 1
4  print(y)

敘述覆蓋：

TC: x=0
  
執行：1→2→3→4
覆蓋：100%敘述 ✓
  
但是：
  分支覆蓋：50%
  因為只測了x=0的分支
  沒測x≠0的分支

分支覆蓋：

TC1: x=0
  路徑：1→2→3→4
  分支：2→3（真）
  
TC2: x=5
  路徑：1→2→4
  分支：2→4（假）
  
兩個測試：
  敘述覆蓋：100% ✓
  分支覆蓋：100% ✓

圖形說明

狀況1：if沒有else

if (條件)
  敘述A
敘述B

覆蓋分析：

敘述覆蓋100%：
  只需1個測試（條件=真）
  執行：條件判斷、敘述A、敘述B
  
分支覆蓋100%：
  需要2個測試
  TC1（條件=真）：條件→A→B
  TC2（條件=假）：條件→B

狀況2：if-else完整

if (條件)
  敘述A
else
  敘述B

覆蓋分析：

敘述覆蓋100%：
  需要2個測試（因為A和B互斥）
  
分支覆蓋100%：
  也需要2個測試
  
此時：
  100%敘述 = 100%分支 ✓

08 13.6 控制流程圖（CFG）

什麼是控制流程圖

定義：

顯示程式碼執行可能路徑的圖形表示

組成元素：

○ 節點（Node）：
  代表程式碼區塊或決策點
  
→ 邊（Edge）：
  代表控制流的轉移（分支）

流程圖 vs 控制流程圖

流程圖：

使用：
  ◇ 菱形 = 決策點
  □ 方形 = 處理步驟
  
範例：
      ┌──────┐
      │讀取數據│
      └───┬──┘
          ↓
       ◇ x>0?
      ╱      ╲
    是         否
    ↓           ↓
  ┌────┐    ┌────┐
  │z=1 │    │z=0 │
  └────┘    └────┘

控制流程圖：

使用：
  ○ 小圓圈 = 節點
  → 箭頭 = 分支
  
範例：
  ○1 (read x)
   ↓
  ○2 (if x>0)
   ↓↘
  ○3  ○4
  (z=1)(z=0)

考試注意事項

兩種圖都可能出現：

✓ 流程圖（傳統）
✓ 控制流程圖（較新）
  
解題方式相同：
  數節點、數分支、追路徑

簡化程式碼：

考試給的程式碼：
  通常是虛擬碼（Pseudocode）
  不是特定語言
  
目的：
  降低語言障礙
  聚焦測試概念

09 13.7 敘述測試實戰 ⭐⭐⭐

基本原則

核心觀念 ⭐⭐⭐：

不要想太複雜！
  
敘述測試只問：
  這行有沒有執行到？
  
不問：
  ✗ 執行幾次
  ✗ 所有分支都執行嗎
  ✗ 邏輯正確嗎

計算步驟

Step 1：數可執行敘述

1  read(x)          ← 算（第1行）
2  read(y)          ← 算（第2行）
3  if (x > y)       ← 算（第3行）
4    z = 0          ← 算（第4行）
5  print(z)         ← 算（第5行）
   
總數：5行

Step 2：追蹤測試執行

TC: x=10, y=5

執行過程：
  行1 ✓ read(x) = 10
  行2 ✓ read(y) = 5
  行3 ✓ if (10 > 5) → 真
  行4 ✓ z = 0
  行5 ✓ print(0)
  
覆蓋：5/5 = 100%

常見陷阱 ⚠️

陷阱1：if判斷式一定會執行

if (x > y)    ← 這行一定會執行！
  z = 0       ← 這行可能不執行

正確觀念：

if (條件)這一行：
  進入判斷時就會執行
  不管條件真假
  
所以100%敘述覆蓋：
  if判斷式必被執行

陷阱2：多個if都要考慮

if (x > 10)
  A
if (x > 20)
  B

解析：

需要測試能執行到A或B：
  
TC: x=15
  執行：第一個if的A
  但未執行：第二個if的B
  
若兩個if內部都要執行：
  可能需要2個測試案例

10 13.8 分支測試實戰 ⭐⭐⭐

識別分支

方法：看箭頭

控制流程圖中：
  每個箭頭 = 一個分支
  
計算分支數 = 數箭頭

分支覆蓋率計算

範例1：基本結構

1  read(x)
2  read(y)
3  if (x > y)
4    z = 0
5  print(z)

控制流程圖：

  ○1
  ↓
  ○2
  ↓
  ○3
 ↙ ↘
○4  ○5
 ↘ ↙
  ○6

分支分析：

1→2：無條件分支
2→3：無條件分支
3→4：條件分支（x>y真）
3→5：條件分支（x>y假）
4→6：無條件分支
5→6：無條件分支

總分支數：6

測試：

TC1: x=10, y=5
  路徑：1→2→3→4→6
  覆蓋：5個分支
  覆蓋率：5/6 ≈ 83%
  
TC2: x=3, y=8
  路徑：1→2→3→5→6
  覆蓋：5個分支
  覆蓋率：5/6 ≈ 83%
  
但注意：
  兩個測試有重疊（1→2）
  
合併覆蓋：6/6 = 100% ✓

分支測試的強度

比敘述測試更強：

原因：
  ✓ 確保每個決策的兩個結果都測到
  ✓ 覆蓋更完整的執行路徑
  ✓ 更容易發現邏輯錯誤

範例：相同測試，不同Bug

正確程式碼：
  if (x > 0)
    y = x + z
  print(y)
  
錯誤程式碼：
  if (x > 0)
    y = x * z    ← 錯誤：應該是+不是*
  print(y)

測試結果：

TC1: x=2, z=3, 期望y=5
  
正確：y = 2+3 = 5 ✓
錯誤：y = 2*3 = 6 ✗
  
敘述覆蓋100%：
  但只用TC1（x>0），未測x≤0
  只達成50%分支覆蓋
  
分支覆蓋100%：
  TC1: x=2, z=3
  TC2: x=0, z=3（測另一分支）
  更全面！

11 13.9 白箱測試的價值與限制

價值 ⭐⭐⭐

1. 客觀的覆蓋度量

優點：

✓ 可精確計算覆蓋率
✓ 有明確的百分比數據
✓ 提供測試充分性的證據

應用：

向客戶證明：
  "我們達到95%敘述覆蓋率"
  
比說：
  "我們測得很仔細"
  更有說服力

2. 獨立於規格

優點：

即使規格：
  ✗ 不完整
  ✗ 過時
  ✗ 錯誤
  
仍可進行白箱測試
  
因為：
  只看程式碼結構

適用場景：

✓ 規格缺失時
✓ 維護老舊系統
✓ 第三方程式碼

3. 容易自動化

原因：

工具可以：
  ✓ 分析程式碼結構
  ✓ 追蹤執行路徑
  ✓ 自動計算覆蓋率
  
不需人工：
  計算覆蓋項目

常見工具：

✓ JUnit (Java)
✓ pytest-cov (Python)
✓ Istanbul (JavaScript)
✓ Coverage.py (Python)

4. 檢測細節缺陷

優勢：

可找到黑箱測試難發現的：
  ✓ 未使用的變數
  ✓ 永遠不會執行的程式碼
  ✓ 邏輯錯誤
  ✓ 過於複雜的結構

限制 ⭐⭐⭐

限制1：無法發現遺漏功能 ⚠️⚠️⚠️

關鍵弱點：

整個功能沒做 → 白箱測試找不到

範例：

需求：
  白金會員折扣20%
  金級會員折扣10%
  其他無折扣
  
程式碼：
  if (金級)
    折扣 = 10%
  else
    折扣 = 0%
    
白箱測試：
  100%覆蓋率 ✓
  
但是：
  白金會員功能完全沒實作！
  黑箱測試才能發現這個問題

教訓：

白箱測試：
  只對"已寫的程式碼"做測試
  
無法驗證：
  該寫但沒寫的功能

限制2：100%覆蓋≠沒Bug ⚠️⚠️⚠️

數學證明範例：

正確程式碼：
  result = a + b

錯誤程式碼：
  result = a * b    ← Bug：應該是+

測試：

TC: a=2, b=2
  
正確：2+2=4
錯誤：2*2=4
  
結果：兩個都得到4！
  
TC通過 ✓
敘述覆蓋100% ✓
分支覆蓋100% ✓
  
但程式碼有Bug！

結論：

即使：
  ✓ 100%敘述覆蓋
  ✓ 100%分支覆蓋
  
也不保證：
  ✗ 系統沒有Bug
  
需要：
  黑箱+白箱並用

限制3：對測試數據不敏感

範例：除零錯誤

result = a / b

測試：

TC1: a=10, b=2
  結果：5
  覆蓋：100% ✓
  
TC2: a=10, b=0
  結果：錯誤！除零異常
  覆蓋：100% ✓
  
兩個測試覆蓋率相同：
  但效果完全不同
  
敘述測試：
  無法區分測試數據的好壞

12 13.10 白箱測試的應用 ⭐⭐

最適用層級：單元測試

為什麼？

單元測試特性：
  ✓ 程式碼小，容易分析
  ✓ 開發者執行，熟悉程式碼
  ✓ 自動化執行
  ✓ 快速回饋
  
白箱測試優勢：
  ✓ 精確覆蓋度量
  ✓ 容易工具化
  ✓ 客觀評估

與黑箱測試互補

互補關係：

黑箱測試：
  ✓ 發現功能遺漏
  ✓ 驗證需求實現
  ✗ 不知內部覆蓋
  
白箱測試：
  ✓ 確保程式碼覆蓋
  ✓ 找到結構問題
  ✗ 發現不了功能遺漏
  
結合使用：
  黑箱設計測試案例
  白箱評估覆蓋充分性
  經驗補充特殊情況

實務做法：

Step 1：用黑箱技術設計測試
  （基於需求）
  
Step 2：執行測試，測量白箱覆蓋率
  （程式碼覆蓋）
  
Step 3：若覆蓋率不足
  分析：哪些路徑未覆蓋
  補充：針對性增加測試
  
Step 4：用經驗導向補充
  （特殊情況）

13 13.11 ISO標準的白箱技術 ⭐

14 13.12 考試題型與解題技巧 ⭐⭐⭐

題型1：覆蓋率計算（K2）

標準題型：

給定程式碼和測試案例
問：敘述/分支覆蓋率是多少？

解題步驟：

Step 1：數總敘述/分支數（分母）

Step 2：追蹤測試執行路徑

Step 3：數執行到的敘述/分支（分子）

Step 4：計算百分比

範例題：

程式碼：
  1  x = 5
  2  y = 10
  3  if (x > 0)
  4    z = x + y
  5  print(z)
  
問：TC (x=5) 的敘述覆蓋率？

解答：
  總敘述：5行
  執行到：1,2,3,4,5（全部）
  覆蓋率：5/5 = 100%

題型2：最少測試案例數（K3）

標準題型：

給定程式碼/流程圖
問：100%覆蓋需要幾個測試案例？

解題技巧：

敘述測試：
  通常1-2個測試案例即可
  
分支測試：
  看有幾個決策點
  每個決策至少2個分支

範例：

程式碼有2個if（沒有else）：
  
最少需要？2個測試案例
  TC1：兩個if都走真分支
  TC2：兩個if都走假分支
  
這樣可達100%分支覆蓋

題型3：定義題（K1）

常考問題：

1. 什麼是白箱測試？
   → 基於內部結構的測試
   
2. 什麼是分支？
   → 控制的轉移
   
3. 敘述覆蓋與分支覆蓋關係？
   → 100%分支 → 100%敘述（反之不成立）

15 13.13 實戰練習題 ⭐⭐⭐

練習1：基本敘述覆蓋

題目：

1  input(A)
2  if (A == 5)
3    B = 4
4  else
5    B = 1
6  print(B)

問題：

TC1: A=5，覆蓋率是？
TC2: A=10，覆蓋率是？
合併覆蓋率是？

解答：

可執行敘述：6行（1,2,3,5,6，4是else關鍵字）
實際算：1,2, (3或5二選一), 6 = 4行

TC1 (A=5)：
  執行：1,2,3,6
  覆蓋：4/4 = 100%
  
TC2 (A=10)：
  執行：1,2,5,6
  覆蓋：4/4 = 100%
  
任一測試都可達100%

練習2：分支覆蓋

題目：

1  read(X)
2  read(Y)
3  if (X > 10)
4    if (Y > 20)
5      Z = 1
6  print(Z)

問題：100%分支覆蓋需要幾個測試？

解答：

分支分析：
  第一個if：
    3→4（X>10真）
    3→6（X>10假）
    
  第二個if：
    4→5（Y>20真）
    4→6（Y>20假）
    
總分支：4個

測試設計：
  TC1: X=15, Y=25 → 執行3→4→5→6
  TC2: X=15, Y=10 → 執行3→4→6
  TC3: X=5, Y=25  → 執行3→6
  
需要3個測試案例

練習3：覆蓋率陷阱

題目：

測試案例T1達到40%敘述覆蓋
測試案例T2達到65%敘述覆蓋

問：下列何者必然為真？

選項分析：

A. T1+T2達到105%覆蓋 ✗
   （覆蓋率不可能>100%）
   
B. 至少有5%的敘述被T1和T2都執行 ✓
   （否則會超過100%）
   
C. 至少5%的敘述不可執行 ✗
   （無法從覆蓋率推論）
   
D. T1+T2達到100%分支覆蓋 ✗
   （敘述覆蓋不保證分支覆蓋）
   
答案：B

16 13.14 重點整理 ⭐⭐⭐

必背定義

敘述測試：

目標：執行每一行可執行敘述
公式：執行敘述數/總敘述數×100%
特性：最弱的白箱技術

分支測試：

目標：執行每一個分支
公式：執行分支數/總分支數×100%
特性：比敘述測試更強

關鍵定理：

100%分支覆蓋 → 100%敘述覆蓋 ✓
100%敘述覆蓋 → 100%分支覆蓋 ✗

優點 vs 缺點

優點：

✓ 客觀的覆蓋度量
✓ 獨立於規格文件
✓ 容易自動化實現
✓ 可檢測結構問題
✓ 提供測試充分性證據

缺點：

✗ 無法發現遺漏功能
✗ 100%覆蓋≠沒有Bug
✗ 對測試數據不敏感
✗ 需要程式碼知識

實務應用

最佳使用時機：

✓ 單元測試（必用）
✓ 元件整合測試
✓ 評估測試充分性
✓ 規格不完整時

與其他技術搭配：

黑箱測試：設計測試案例（基於需求）
白箱測試：評估覆蓋充分性（基於程式碼）
經驗測試：補充特殊情況（基於經驗）

必記事項

5個核心知識點：

考試技巧

技巧1：快速判斷

看到程式碼題目：
  
第一步：先判斷題型
  敘述測試？分支測試？
  
第二步：數分母
  有幾行/幾個分支？
  
第三步：追路徑
  測試案例執行哪些？
  
第四步：算比例
  分子/分母×100%

技巧2：if判斷必執行

常見錯誤：
  以為if(條件)這行可能不執行
  
正確觀念：
  if(條件)一定會執行
  只有內部程式碼才依條件執行

技巧3：看圖做答

給控制流程圖：
  數箭頭 = 分支數
  走一遍 = 算覆蓋
  
比看程式碼快！
