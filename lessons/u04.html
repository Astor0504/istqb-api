03 4.1 測試活動概述

活動、流程與目的

基本概念

目的（Purpose）
    ↓
需要流程（Process）來達成
    ↓
流程由多個活動（Activity）組成

例子：

  • 目的：完成高品質的軟體測試
  • 流程：測試流程
  • 活動：規劃、分析、設計、建置、執行、監控、完成

測試的七大活動

完整列表

  1. 測試規劃（Test Planning）
  2. 測試分析（Test Analysis）
  3. 測試設計（Test Design）
  4. 測試建置（Test Implementation）
  5. 測試執行（Test Execution）
  6. 測試監控（Test Monitoring and Control）⚡
  7. 測試完成（Test Completion）

⚡ 特殊說明：監控是持續進行，不在排序選項中

活動執行方式

1. 循序執行（Sequential）

規劃 → 完成 → 分析 → 完成 → 設計 → 完成
  • 一個活動完全結束才開始下一個
  • 適合：瀑布式開發

2. 迭代執行（Iterative）

快速輪轉：規劃→分析→設計→建置→執行
           ↓
        第二輪：規劃→分析→設計...
  • 多次快速循環
  • 適合：敏捷開發

3. 重疊執行（Overlapping）

規劃 ████████
      分析 ████████
            設計 ████████
  • 前一個活動未完成，下一個已開始
  • 最靈活的方式

考試重點 ⭐

排序題（必考）

題型：請排列測試活動的正確順序

標準答案順序（6個）：

  1. 測試規劃
  2. 測試分析
  3. 測試設計
  4. 測試建置
  5. 測試執行
  6. 測試完成

注意：監控不會出現在排序選項（因為持續進行）

易混淆點

  • ❌ 設計 → 分析（順序錯誤）
  • ✓ 分析 → 設計（正確順序）

04 4.2 測試規劃（Test Planning）

活動定義

確定測試目的、範圍、策略和資源

主要工作

1. 定義測試範圍

  • 要測試什麼？
  • 不測試什麼？
  • 測試的深度？

2. 確定測試目標

  • 找缺陷？
  • 建立信心？
  • 評估風險？

3. 規劃資源

  • 需要多少人？
  • 需要什麼工具？
  • 需要什麼環境？

4. 制定策略

  • 用什麼測試技術？
  • 如何安排優先順序？
  • 如何管理風險？

重要產物：測試計劃書 📋

ISO 29119 標準欄位（9個）

#欄位名稱說明重要性
1測試時程時間安排⭐⭐⭐
2引入標準進入條件⭐⭐⭐
3引出標準退出條件⭐⭐⭐
4風險清單產品風險⭐⭐⭐
5測試範圍涵蓋範圍⭐⭐
6測試策略測試方法⭐⭐
7測試活動估計工作量⭐
8人力資源人員分配⭐
9度量指標衡量標準⭐⭐

記憶技巧

核心4欄位（必背）：

  1. 測試時程（Schedule）
  2. 引入/引出標準（Entry/Exit Criteria）
  3. 風險清單（Risk Register）
  4. （其他欄位都在測試計劃書內）
測試計劃書
├── 時程表
├── 引入/引出標準
├── 風險清單
└── 其他相關欄位

規劃不是一次性的 ⚠️

持續調整

規劃 → 監控 → 發現問題 → 調整規劃
  ↑                          ↓
  └──────────────────────────┘
         持續回饋修正

錯誤觀念：規劃一次就定案 正確觀念：規劃是持續性活動

05 4.3 測試監控與控制（Test Monitoring and Control）

兩大組成

監督（Monitoring）

目的：收集資料，了解現況

收集內容：

  • 測試進度如何？
  • 發現多少缺陷？
  • 風險狀況如何？
  • 覆蓋率達到多少？

控制（Control）

目的：採取行動，改善品質

控制措施：

  • 延長測試時間
  • 增加測試資源
  • 調整測試範圍
  • 變更測試策略

重要產物：測試進度報告 📊

測試進度報告欄位（6-8個，必背）

第五章會詳細說明

重要性：

  • 向利害關係人溝通測試狀態
  • 提供決策所需資訊
  • 追蹤測試進展

報告方式

  • ✓ 書面報告
  • ✓ 口頭報告
  • ✓ Dashboard（儀表板）
  • ✓ 會議簡報

控制指示文件

用途：記錄控制決策

重要性：

  • 留下證據（便於日後對質）
  • 不只是口頭承諾
  • 文件化管理決策

例子：

因測試覆蓋率僅達 60%，決定延長測試兩週， 並增派 2 名測試人員。

06 4.4 測試分析（Test Analysis）

活動定義

分析測試依據，找出測試條件

兩大步驟

步驟1：審查測試依據

測試依據（Test Basis）：

任何可以說明受測系統運作的資訊或文件

包含：

  • 需求規格書
  • Use Case（使用案例）
  • User Story（使用者故事）
  • 設計文件
  • 程式碼
  • 商業規則
  • 甚至示意圖

審查目標：

  • ✓ 完整性（Completeness）
  • ✓ 一致性（Consistency）
  • ✓ 準確性（Accuracy）
  • ✓ 可測試性（Testability）

問題檢查：

  • ❌ 內容相互衝突
  • ❌ 遺漏重要資訊
  • ❌ 含糊不清
  • ❌ 前後不一致

步驟2：分析測試條件

測試條件（Test Condition）：

可被測試的觀點或面向

從測試依據中找出：

  • 要測試什麼功能？
  • 要測試什麼品質特徵？
  • 要涵蓋哪些情境？

重要概念：正確答案在哪裡？ 🎯

核心邏輯

規格書（測試依據）
    ↓
定義了正確答案
    ↓
開發依規格實作
    ↓
測試依規格驗證

例子：系統要不要有歡迎畫面？

  • ✓ 規格有寫 → 要有
  • ✗ 規格沒寫 → 不要求
  • ❌ 測試人員覺得要有 → 不算（不是正確答案來源）

文件品質的重要性

規格亂寫 → 開發亂做 → 測試亂設計
   ↓
全盤皆亂

規格正確 → 開發正確 → 測試正確
   ↓
步步到位

測試條件範例

登入頁面的測試條件

功能性：

  • 帳號欄位可輸入
  • 密碼欄位可輸入且遮罩
  • 登入按鈕可點擊
  • 正確帳密可登入
  • 錯誤帳密不可登入
  • 忘記密碼功能
  • 語言切換功能

非功能性：

  • 速度（載入時間 < 3秒）
  • 安全性（密碼加密）
  • 使用性（介面友善）
  • 相容性（支援多瀏覽器）
  • 易學性（容易上手）

測試條件排序 🏆

為什麼要排序？

  1. 測試條件很多
  2. 時間有限
  3. 需要優先順序

排序依據

  • 重要性
  • 風險高低
  • 業務價值
  • 技術依賴

ISO 25010 品質特徵（9個必背）⭐

#品質特徵英文說明
1功能適用性Functional Suitability功能完整、正確、適用
2效能效率Performance Efficiency速度、資源使用
3相容性Compatibility與其他系統共存
4易用性Usability好不好用
5可靠性Reliability穩定、可用
6安全性Security保護資訊、防止攻擊
7維護性Maintainability易於修改
8可攜性Portability可移植到其他環境
9安全保障性Safety人身安全（較少用）
記憶口訣：功效相易可安維可安

考試重點：考「以下何者為 ISO 25010 的品質特徵？」

電商網站排序範例

假設需求：開發電子商務網站

排序（依重要性）：

  1. 速度（效能效率）→ 5秒內載入，否則使用者流失
  2. 易用性（Usability）→ 好用才有人買
  3. 安全性（Security）→ 保護交易資料
  4. 相容性（Compatibility）→ 手機/PC 都能用
  5. 易學性（Usability子項）→ 新手也會用
  6. 維護性（Maintainability）→ 最後才考慮

結論：

  • 功能性：必須要有（基本要求）
  • 非功能性：決定競爭力

主要產物

  1. 已排序的測試條件（最重要）
  2. 驗收標準（Acceptance Criteria）
  3. 缺陷報告（針對測試依據的問題）

07 4.5 測試設計（Test Design）

活動定義

將測試條件細化為測試案例

主要工作

1. 設計測試案例

從測試條件 → 具體的測試案例

測試案例結構（必背）⭐：

欄位說明必要性
UID唯一識別碼必要
目標測試目的必要
優先性高/中/低必要
可追溯性對應的需求/風險必要
前置條件測試前提必要
輸入步驟 + 數據必要
期望輸出正確答案必要
實際輸出系統反應執行時填

2. 識別覆蓋項目

  • 依測試技術而定
  • 第四章會詳細說明

3. 定義測試數據需求

  • 需要什麼測試資料？
  • 資料從哪裡來？

高層級 vs 低層級測試案例

低層級測試案例

包含：步驟 + 具體數據

例子：

步驟1：輸入帳號 "test001"
步驟2：輸入密碼 "Pass@123"
步驟3：點擊登入按鈕
期望：成功登入，顯示首頁

特點：

  • ✓ 任何人執行結果一致
  • ✓ 容易自動化
  • ✗ 較缺乏彈性

高層級測試案例

包含：步驟（無具體數據）

例子：

步驟1：輸入有效帳號
步驟2：輸入有效密碼
步驟3：點擊登入按鈕
期望：成功登入

特點：

  • ✓ 可重用性高
  • ✓ 測試人員可發揮專業
  • ✗ 不同人執行可能結果不同

使用時機

高層級：

  • 系統前期（需求未明確）
  • 需要重用的測試
  • 測試人員經驗豐富

低層級：

  • 需求明確
  • 準備自動化
  • 確保一致性

主要產物

  1. 測試案例（最重要）
  2. 測試章程（Test Charter）- 用於探索性測試
  3. 覆蓋項目
  4. 測試數據需求
  5. 測試環境需求

08 4.6 測試建置（Test Implementation）

活動定義

準備測試執行所需的一切資源

主要工作

1. 建立測試環境

軟體：

  • 測試伺服器
  • 資料庫
  • 應用程式
  • 測試工具

硬體：

  • 測試機器
  • 網路設備
  • 行動裝置

2. 準備測試數據

  • 建立測試帳號
  • 準備測試資料
  • 設定初始狀態

3. 建立測試程序

  • 組織測試案例
  • 撰寫測試腳本
  • 安排執行順序

4. 建立測試套件

  • 分組測試案例
  • 建立迴歸測試套件
  • 建立冒煙測試套件

5. 準備測試工具

  • 安裝測試工具
  • 設定工具環境
  • 驗證工具可用

主要產物

  1. 測試程序（Test Procedures）
  2. 自動化測試腳本
  3. 測試套件（Test Suites）
  4. 測試數據
  5. 測試執行時程表
  6. 測試環境元件（Stubs、Drivers、Simulators）

關鍵成功因素

確保一切準備就緒

09 4.7 測試執行（Test Execution）

活動定義

實際執行測試案例

主要工作

1. 執行測試

  • 手動執行
  • 自動化執行
  • 依測試程序執行

2. 比對結果

期望輸出（正確答案）
      ↕ 比對
實際輸出（系統反應）
      ↓
   相同 → 通過 ✓
   不同 → 失敗 ✗

3. 記錄測試結果

  • 通過的測試案例
  • 失敗的測試案例
  • 發現的缺陷

4. 分析失敗原因

  • 是否為真實缺陷？
  • 缺陷的嚴重程度？
  • 可能的成因？

5. 執行迴歸測試

修改後，重新執行相關測試

主要產物

1. 測試日誌（Test Log）⭐

記錄內容：

  • 執行哪些測試案例
  • 執行時間
  • 執行者
  • 測試結果

2. 缺陷報告（Defect Report）⭐⭐⭐

欄位（第五章會詳細說明，需背誦）：

必要欄位：

  • 缺陷 ID
  • 標題
  • 描述
  • 重現步驟
  • 期望結果
  • 實際結果
  • 嚴重性（Severity）
  • 優先性（Priority）
  • 狀態（Status）
  • 測試環境

重要提醒：

  • 要能讓開發人員重現問題
  • 提供足夠資訊（但不過多）
  • 附上截圖或log 檔

10 4.8 測試完成（Test Completion）

活動定義

收集測試成果，進行總結學習

主要工作

1. 收集測試產物

  • 整理測試文件
  • 歸檔測試案例
  • 保存測試數據

2. 撰寫測試總結報告

3. 經驗學習

  • 什麼做得好？
  • 什麼需改進？
  • 學到什麼教訓？

4. 歸檔與移交

  • 文件歸檔
  • 環境復原
  • 產物移交

5. 處理剩餘問題

  • 未修復的缺陷
  • 納入產品待辦清單
  • 規劃下次處理

主要產物

測試總結報告（Test Completion Report）⭐⭐

欄位（第五章會詳細說明，需背誦）

用途：

  • 總結測試成果
  • 提供利害關係人資訊
  • 記錄經驗教訓

傳達對象：

  • 管理層
  • 客戶
  • 專案團隊
  • 維護團隊

12 4.10 可追溯性（Traceability）

定義

建立工作產物之間的關聯

雙向可追溯性 ⭐ 重要

前向追溯（Forward Traceability）

需求 REQ-001
    ↓ 對應
測試案例 TC-001, TC-002, TC-003

問題：這個需求有哪些測試案例？ 答案：查表可得

反向追溯（Backward Traceability）

測試案例 TC-001
    ↑ 來自
需求 REQ-001

問題：這個測試案例測試哪個需求？ 答案：查表可得

矩陣範例

REQ-001REQ-002REQ-003
TC-001X
TC-002XX
TC-003XX
TC-004X

分析：

  • REQ-001 被 TC-001, TC-002 覆蓋
  • REQ-002 被 TC-002, TC-003 覆蓋
  • REQ-003 被 TC-003, TC-004 覆蓋
  • 所有需求都有測試覆蓋 ✓

可追溯性的價值

1. 變更影響分析

情境：需求 REQ-002 修改

分析：

  • 查可追溯性矩陣
  • 發現影響 TC-002, TC-003
  • 只需重新執行這兩個測試案例

2. 覆蓋率分析

檢查：

  • 哪些需求沒有測試案例？
  • 哪些風險沒有被覆蓋？

3. 改善溝通

好處：

  • 清楚的對應關係
  • 便於討論
  • 減少誤解

4. 支援 IT 治理

13 4.11 測試角色

兩大角色分類

測試管理角色（Test Management Role）

聚焦活動：

  1. 測試規劃
  2. 測試監控與控制
  3. 測試完成

主要職責（需理解）：

  • 制定或檢討測試策略與政策
  • 風險管理
  • 資源規劃與獲取
  • 報告測試進度
  • 監控測試流程
  • 提升測試團隊地位

測試角色（Testing Role）

聚焦活動：

  1. 測試分析
  2. 測試設計
  3. 測試建置
  4. 測試執行

主要職責（需理解）：

  • 協作撰寫使用者故事
  • 產生可測試的驗收標準
  • 分析、審查測試依據的可測試性
  • 設計、實作測試案例
  • 準備測試環境
  • 執行測試
  • 記錄測試結果

角色與人員的關係

一個人可以扮演多種角色

例子1：小團隊

測試工程師 Alice
├── 測試管理角色（撰寫測試計劃）
└── 測試角色（設計並執行測試）

例子2：大團隊

測試經理 Bob → 測試管理角色
測試工程師 Carol → 測試角色
測試工程師 Dave → 測試角色

角色也可以兼職

例子：

開發經理
├── 開發管理（主要）
└── 測試管理角色（兼任）

考試題型 ⭐

題型1：角色識別

問法：「以下何者為測試管理角色的工作？」

選項混雜：

  • 3 個測試管理角色工作
  • 1 個測試角色工作 ← 答案（何者為「非」）

作答技巧：

  • 記住兩種角色的核心職責
  • 用聯想記憶（不要死背）

重要提醒

每家公司的角色定義可能不同 考試以官方定義為準
