01 單元16：測試估計技術

  • 掌握基於專家的估計技術

03 16.1 測試估計的挑戰 ⭐⭐

為什麼估計困難？

現實問題：

估計總是不準！
  
為什麼？
  ✗ 需求不明確
  ✗ 技術不確定
  ✗ 人為因素
  ✗ 環境變化
  
結果：
  估計偏差大

估計的重要性

為什麼還要估計？ ⭐⭐

引用名言：
  "在準備戰鬥時，我意識到計劃是沒有用的，
   但計劃是必不可少的。"
   - 艾森豪將軍
   
意思：
  計劃趕不上變化 ✓
  但計劃仍然必要 ✓
  
核心價值：
  強迫提前思考問題

好的估計能帶來什麼？

✓ 資源規劃
✓ 進度安排
✓ 風險識別
✓ 決策依據
✓ 利害關係人溝通

任務拆解原則 ⭐⭐

大任務難估計：

問題：
  估算一台車值多少錢？
  → 很難！
  
解決：
  拆解成零件估計：
  - 引擎多少錢
  - 輪胎多少錢
  - 車身多少錢
  - ...
  
加總 = 總價
  
更準確！

原則：

Step 1：拆解大任務
  ↓
Step 2：估計小任務
  ↓
Step 3：加總
  ↓
Step 4：加入緩衝

04 16.2 估計技術的分類 ⭐⭐⭐

兩大類（必背）

分類依據：

依據什麼來估計？

類別1：基於度量指標
  → 需要資料（Data）
  
類別2：基於專家意見
  → 需要人（Expert）

對應技術 ⭐⭐⭐（必背）：

基於度量指標：
  1. 比率估計 (Ratio Estimation)
  2. 推估法 (Extrapolation)
  
基於專家意見：
  1. 廣義德菲法 (Wide Band Delphi)
  2. 三點估計 (Three-Point Estimation)

考試重點 ⭐⭐⭐：

題型1：
  下列何者為基於度量指標的估計技術？
  答案：比率估計、推估法
  
題型2：
  下列何者為基於專家的估計技術？
  答案：廣義德菲法、三點估計

05 16.3 比率估計 ⭐⭐⭐

基本概念

定義：

使用歷史專案的標準比率來估算測試工作量

核心公式：

開發工作量：測試工作量 = X：Y

已知：
  新專案開發工作量 = A
  
求：
  新專案測試工作量 = ?
  
答案：
  = A × (Y/X)

如何取得比率？ ⭐⭐⭐

從歷史資料統計：

統計組織內部歷史專案：

專案    開發工作量    測試工作量
P1      $800,000     $40,000
P2      $1,200,000   $130,000
P3      $600,000     $70,000
P4      $1,000,000   $120,000
────────────────────────────────
平均    $900,000     $90,000

比率 = 測試/開發
     = $90,000 / $900,000
     = 1:10
     = 10%

計算步驟 ⭐⭐⭐：

Step 1：收集歷史資料
  → 開發工作量總和
  → 測試工作量總和
  
Step 2：計算比率
  → 測試/開發
  
Step 3：應用到新專案
  → 新專案開發工作量 × 比率

實戰範例 ⭐⭐⭐

題目：

歷史資料（4個專案）：
  開發總和：$3,600,000
  測試總和：$360,000
  
比率：
  $360,000 / $3,600,000 = 1:10 = 10%
  
新專案：
  開發工作量預估：$800,000
  
問：測試工作量？
  
答案：
  $800,000 × 10% = $80,000

注意事項 ⭐⭐：

✓ 必須是相似專案
  （ERP vs ERP，不能用Web vs 嵌入式）
  
✓ 比率會更準確
  （專案類型相同）
  
✗ 跨領域比率可能不準

優缺點

優點：

✓ 有歷史數據支持
✓ 計算簡單
✓ 易於理解
✓ 客觀

缺點：

✗ 需要歷史資料
✗ 專案特性要相似
✗ 不適用於新技術
✗ 無法反映創新

06 16.4 推估法 ⭐⭐⭐

基本概念

定義：

基於當前專案已知數據來預測未來

核心思想：

用已知值 → 擴展未知值

適用情境 ⭐⭐

最適合：

迭代式SDLC
  
為什麼？
  每次迭代都有實際數據
  可用來預測下次迭代

限制：

✗ 第一次迭代無法使用
  （沒有歷史數據）
  
解決方法：
  ✓ 用組織歷史數據
  ✓ PO粗估
  ✓ 用比率估計

團隊速度（Velocity）⭐⭐⭐

定義：

團隊在一次迭代中可完成的工作量

計算方式：

迭代#1：完成 38 人天的工作
迭代#2：完成 29 人天的工作
迭代#3：完成 38 人天的工作

預測迭代#4：
  = (38 + 29 + 38) / 3
  = 105 / 3
  = 35 人天

一般公式：

預估第n次迭代：
  
E(n) = 平均前幾次迭代的實際工作量

或更精確：
E(n) = (3×A(n-1) + A(n-2)) / 4

其中：
  E(n) = 第n次迭代的預估
  A(n-1) = 第n-1次迭代的實際
  A(n-2) = 第n-2次迭代的實際

實戰範例 ⭐⭐⭐

範例1：簡單平均

過去3次迭代：
  Iteration 1：40 story points
  Iteration 2：35 story points
  Iteration 3：45 story points
  
預估Iteration 4：
  = (40 + 35 + 45) / 3
  = 40 story points

範例2：加權平均

已知：
  A(4) = 6 人天
  A(3) = 8 人天
  
使用公式：
  E(5) = (3×A(4) + A(3)) / 4
       = (3×6 + 8) / 4
       = (18 + 8) / 4
       = 26 / 4
       = 6.5 人天

優缺點

優點：

✓ 基於實際數據
✓ 反映團隊能力
✓ 持續修正
✓ 越來越準

缺點：

✗ 第一次無法用
✗ 需要時間累積
✗ 團隊變動會影響

07 16.5 廣義德菲法與規劃撲克 ⭐⭐⭐

傳統德菲法

特點：

✓ 專家匿名
✓ 多輪估計
✓ 不見面
✓ 透過問卷
  
問題：
  ✗ 耗時很長
  ✗ 專家不改意見
  ✗ 溝通困難

廣義德菲法 ⭐⭐⭐

改進：

✓ 專家非匿名
✓ 多輪估計
✓ 面對面討論
✓ 達成共識
  
特點：
  專家可以看到彼此
  聚在一起討論

流程：

Round 1：各自獨立估計
  ↓
顯示結果
  ↓
討論差異
  ↓
Round 2：再次獨立估計
  ↓
顯示結果
  ↓
討論差異
  ↓
持續...直到達成共識

規劃撲克（Planning Poker）⭐⭐⭐

定義：

廣義德菲法在敏捷的變形

使用卡牌：

費氏數列（變形）：
  0, 1, 2, 3, 5, 8, 13, 21, 40, 100, ?, ☕
  
為什麼用費氏數列？
  ✓ 數字間差距遞增
  ✓ 反映不確定性
  ✓ 避免過度精確

為什麼變形？ ⭐⭐

標準費氏：
  1, 2, 3, 5, 8, 13, 21, 34, 55...
  
變形費氏：
  1, 2, 3, 5, 8, 13, 21, 40, 100
  
為什麼在21後跳到40？
  
理由：
  任務超過21天已經很複雜
  不可能估得那麼準
  沒必要區分34和55
  
  超過21 = 要拆分

特殊卡牌：

?（問號）卡：
  → 完全不知道，需要更多資訊
  
☕（咖啡）卡：
  → 需要休息一下

規劃撲克流程 ⭐⭐⭐

詳細步驟：

Step 1：PO讀使用者故事
  ↓
Step 2：團隊討論、提問
  PO回答、釐清期望
  ↓
Step 3：每人獨立選卡
  （不讓別人看到）
  ↓
Step 4：同時翻牌
  
結果1：所有人相同數字
  → 達成共識，記錄答案
  
結果2：數字不同
  → 進入討論
  ↓
Step 5：請最小和最大值的人說明
  "為什麼你認為是X？"
  ↓
Step 6：全員聽完後，再次獨立估計
  ↓
Step 7：重複直到達成共識

實戰範例：

Round 1：3, 8, 2, 5
  → 沒有共識
  → 請最小(2)和最大(8)說明理由
  
測試者(8)說：
  "這個很難測試，需要建立新的測試框架"
  
開發者(2)說：
  "我以前寫過類似的，不會太難"
  
Round 2：5, 5, 5, 8
  → 還是沒完全共識
  → 再請8說明
  
Round 3：5, 5, 5, 5
  → 達成共識！
  → 記錄：5 story points

如何開始？ ⭐⭐

建立基準：

問題：
  第一次怎麼估？沒有參考
  
建議：
  不要先玩撲克
  
  先找一個故事定為「2」
  再找一個故事定為「5」
  
基準建立：
  有了2和5
  其他故事就可以比較
  
範例：
  這個比「2」大，但比「5」小
  → 可能是3

共識規則 ⭐⭐

何時算達成共識？

理想：
  所有人數字相同
  
實務：
  可設定規則：
  ✓ 80%的人同意 = 共識
  ✓ 最大和最小差距≤3 = 共識
  
避免：
  無止境討論

強制收斂：

情況：
  討論多輪仍無共識
  
處理：
  1. 檢查是否產品不確定性
     → 需要收集更多資訊
     
  2. 檢查是否技術不確定性
     → 需要技術研究
     
  3. 差距很大時
     → 記錄為範圍（如：5-100）

優缺點

優點：

✓ 建立團隊共識
✓ 考慮多元觀點
✓ 促進溝通
✓ 趣味性高
✓ 避免主管獨斷

缺點：

✗ 耗時
✗ 需要全員參與
✗ 可能受群體壓力影響

08 16.6 三點估計法 ⭐⭐⭐

基本概念

定義：

結合最樂觀、最可能、最悲觀三個估計值

三個點：

a = 最樂觀估計 (Optimistic)
m = 最可能估計 (Most Likely)
b = 最悲觀估計 (Pessimistic)

核心公式（必背）⭐⭐⭐

期望值公式：

E = (a + 4×m + b) / 6

其中：
  E = 期望值（Expected Value）
  a = 最樂觀
  m = 最可能
  b = 最悲觀

標準差公式：

SD = (b - a) / 6

簡化記憶：
  SD = (最大值 - 最小值) / 6
  
不要記成：
  ✗ (最悲觀 - 最樂觀) / 6
  
因為太繞口！

實戰範例 ⭐⭐⭐

範例1：單一任務

估計單一測試案例執行時間：
  最樂觀：1小時
  最可能：3小時
  最悲觀：8小時
  
計算期望值：
  E = (1 + 4×3 + 8) / 6
    = (1 + 12 + 8) / 6
    = 21 / 6
    = 3.5 小時
    
如果要執行4個測試案例：
  總時間 = 3.5 × 4 = 14 小時

範例2：考試準備

準備ISTQB考試所需時間：
  最樂觀：10天（已有經驗）
  最可能：70天（正常準備）
  最悲觀：90天（完全新手）
  
期望值：
  E = (10 + 4×70 + 90) / 6
    = (10 + 280 + 90) / 6
    = 380 / 6
    = 63.3 天
    
標準差：
  SD = (90 - 10) / 6
     = 80 / 6
     = 13.3 天

如何解讀結果 ⭐⭐⭐

期望值±標準差：

E = 63.3 天
SD = 13.3 天

結果區間：
  E - SD = 63.3 - 13.3 = 50 天
  E + SD = 63.3 + 13.3 = 76.6 天
  
解讀：
  ✓ 合理預估落在 50-77 天之間
  ✓ 最可能是 63 天

信心水準（課外）：

統計學概念：
  
E ± SD：約68%機率
E ± 2SD：約95%機率
E ± 3SD：約99.7%機率
  
但考試不考這麼深！

考試題型 ⭐⭐⭐

常見考法：

題目給你：
  最樂觀：2 人時
  最可能：11 人時
  最悲觀：14 人時
  
問：最終估計？
  
解答：
  E = (2 + 4×11 + 14) / 6
    = (2 + 44 + 14) / 6
    = 60 / 6
    = 10 人時
    
答案：d) 10 人時

解題技巧：

1. 清楚辨識 a, m, b
2. 套用公式
3. 計算時注意：
   - 4×m 要先算
   - 最後除以6

公式記憶訣竅 ⭐⭐⭐

期望值公式：

記憶法：
  "1-4-1除以6"
  
  a + 4m + b
  ─────────
      6
      
為什麼是4？
  因為最可能發生
  所以權重最高

標準差公式：

記憶法：
  "最大減最小除以6"
  
  不要記：
  ✗ 最悲觀減最樂觀
  
  因為：
  萬一題目順序不同
  你會搞混

背後理論（課外）

Beta分配：

三點估計基於統計學的Beta分配
  
但考試不考理論！
  
只要記住公式即可

09 16.7 相對估計 vs 絕對估計 ⭐⭐

絕對估計

定義：

用實際的時間或工作量單位
  
範例：
  ✓ 3天
  ✓ 40小時
  ✓ 2週

問題：

✗ 容易估錯
✗ 受團隊能力影響大
✗ 容易產生承諾壓力

相對估計 ⭐⭐⭐

定義：

相較於參考點的相對大小
  
範例：
  ✓ 這個是2 points
  ✓ 那個是5 points
  ✓ 這個比那個大

Story Points：

不是實際時間
是相對大小
  
範例：
  2 points的故事
  不代表2天完成
  
  只代表：
  是5 points故事的40%大小

敏捷常用：

為什麼敏捷喜歡相對估計？
  
✓ 避免承諾壓力
✓ 聚焦工作複雜度
✓ 不受個人能力影響
✓ 更容易達成共識
  
規劃撲克 = 相對估計工具

10 16.8 估計技術比較 ⭐⭐⭐

四種技術對比

技術          分類        資料需求      適用時機        優點
──────────────────────────────────────────────────────────
比率估計      度量指標    歷史專案      新專案開始      簡單、客觀
推估法        度量指標    當前專案      迭代專案        反映實際
廣義德菲法    專家        無            任何時候        建立共識
三點估計      專家        無            任何時候        考慮風險

選擇建議

新專案：

推薦：
  1. 比率估計（初估）
  2. 廣義德菲法（精估）
  3. 三點估計（考慮風險）

迭代專案：

推薦：
  1. 推估法（基於速度）
  2. 規劃撲克（團隊共識）

時間緊迫：

推薦：
  專家估計（快）

需要精確：

推薦：
  三點估計（有區間）

11 16.9 估計的實務建議 ⭐⭐

估計時的注意事項

1. 不要過度精確：

✗ 壞：23.47 人時
✓ 好：20-25 人時

原因：
  估計本來就不準
  假裝精確沒意義

2. 包含緩衝：

估計：40小時
建議加入：20-30%緩衝
實際報告：50-52小時
  
為什麼？
  ✓ 總有意外
  ✓ 總有遺漏
  ✓ 總有干擾

3. 定期修正：

估計不是一次性
  
應該：
  ✓ 定期檢視
  ✓ 根據實際調整
  ✓ 持續改善

4. 記錄假設：

估計時的假設：
  ✓ 團隊不變
  ✓ 技術棧不變
  ✓ 需求不大改
  
假設改變：
  估計要重做

常見錯誤

錯誤1：只憑感覺

✗ "我覺得3天夠了"
  
✓ 應該：
  基於資料或討論

錯誤2：被老闆估

✗ 老闆說："這個1天搞定！"
  
問題：
  老闆通常不寫程式
  估計會偏離現實
  
✓ 應該：
  團隊估計
  多元觀點

錯誤3：估了不改

✗ 估完就固定
  
✓ 應該：
  持續修正
  反映現實

12 16.10 考試重點整理 ⭐⭐⭐

必背內容

1. 估計技術分類：⭐⭐⭐

基於度量指標：
  ✓ 比率估計
  ✓ 推估法
  
基於專家：
  ✓ 廣義德菲法
  ✓ 三點估計

2. 三點估計公式：⭐⭐⭐

期望值：E = (a + 4m + b) / 6
標準差：SD = (b - a) / 6

記憶：
  ✓ 1-4-1除以6
  ✓ 最大減最小除以6

3. 比率估計計算：⭐⭐⭐
