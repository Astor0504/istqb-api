03 7.1 靜態測試概述 ⭐⭐⭐

定義

不執行軟體，透過檢查工作產品來了解軟體好壞

核心概念

靜態測試：

  • 軟體不用執行
  • 檢查文件、程式碼
  • 早期發現問題

動態測試：

  • 軟體要能執行
  • 實際運行測試
  • 中後期進行

靜態測試的兩大技術 ⭐⭐⭐ 必考

1. 審查（Review）

特性：

  • 手動進行
  • 需要人工參與
  • 群體活動

80% 官方講義內容

2. 靜態分析（Static Analysis）

特性：

  • 工具自動化
  • 程式掃描
  • 快速檢測

20% 官方講義內容

考題：

以下何者為靜態測試技術？ 答案：審查、靜態分析

04 7.2 靜態測試 vs 動態測試 ⭐⭐⭐

共同目標

相同點：

  1. 評估工作產品品質
  2. 儘快找到缺陷

關係：互補

主要差異

項目靜態測試動態測試
執行軟體否是
時機早期中後期
發現方式直接找 Bug透過失敗證明 Bug
成本低（早期）高（晚期）
適用對象文件、程式碼可執行軟體

尋找缺陷的方式

靜態測試：

審查程式碼
  ↓ 直接發現
Bug（程式錯誤）
  ↓ 立即
修正

動態測試：

執行程式
  ↓ 發現
Failure（失敗現象）
  ↓ 證明
Bug 存在
  ↓ 找出
Bug 位置
  ↓ 最後
修正

早期測試的價值 💡

修復成本比較

2002 年調查：

需求階段發現：成本 = 1
發佈階段發現：成本 = 30
→ 差距 30 倍

2010 年調查：

需求階段發現：成本 = 1
發佈階段發現：成本 = 100
→ 差距 100 倍

2025 年現在：

系統更複雜（AI、行動裝置、雲端）
→ 差距可能更大

結論：

越早發現，越便宜修

例子：相同工作產品的靜態與動態測試

工作產品：查詢功能

靜態測試：

審查需求文件：
  ✓ 需求描述完整嗎？
  ✓ 語法結構正確嗎？
  ✓ 是否有矛盾？

靜態分析程式碼：
  ✓ 是否有重複程式碼？
  ✓ 變數是否有未使用？
  ✓ 複雜度是否過高？

動態測試：

執行查詢功能：
  ✓ 輸入商品編號
  ✓ 是否回傳正確商品？
  ✓ 回應時間是否符合要求？
  ✓ 結果是否有排序？

互補關係：

  • 靜態：早期預防
  • 動態：實際驗證

05 7.3 靜態測試的價值 ⭐⭐

主要優點

1. 早期發現缺陷

好處：

  • 修復成本低
  • 避免連鎖影響
  • 減少後期返工

2. 檢測特定類型缺陷

靜態測試較容易發現：

需求相關：

  • 需求不完整
  • 需求矛盾
  • 需求模糊

設計相關：

  • 設計不一致
  • 模組化不佳
  • 耦合度過高

程式碼相關：

✓ 變數未定義
✓ 變數重複定義
✓ 變數宣告未使用
✓ 程式碼無法執行到
✓ 程式碼過於複雜

標準相關：

  • 偏離編碼標準
  • 命名規則不符
  • 參數數量/順序錯誤

安全相關：

  • 簡單的安全漏洞
  • SQL Injection 風險
  • XSS 漏洞

覆蓋度相關：

  • 測試涵蓋不足
  • 需求未被覆蓋

3. 提升品質信心

  • 規格審查通過 → 開發有信心
  • 設計審查通過 → 實作有依據
  • 程式碼審查通過 → 測試更有效

4. 增進理解與溝通

  • 團隊達成共識
  • 減少誤解
  • 改善協作

5. 降低整體成本

雖然：

  • 導入需要成本
  • 需要人力時間

但是：

  • 早期修復便宜
  • 減少後期返工
  • 整體成本降低

實際案例：模擬比較

情境 A：無靜態測試

階段         發現 Bug    修復成本
需求        0           0
設計        0           0
編碼        0           0
測試        50          50 × 100 = 5000
總計        50 bugs     成本：5000

情境 B：有靜態測試

階段         發現 Bug    修復成本
需求審查    15          15 × 1 = 15
設計審查    12          12 × 10 = 120
程式碼審查  18          18 × 25 = 450
測試        5           5 × 100 = 500
總計        50 bugs     成本：1085

靜態測試額外成本：200
總成本：1285

結論：

  • 情境 A：5000
  • 情境 B：1285
  • 節省：74%

06 7.4 審查（Review）概述

定義

一群人手動檢查工作產品

可審查的工作產品 ⭐

原則：

能閱讀理解的都可以審查

包含：

  • 合約
  • 需求規格
  • 設計文件
  • 原始碼
  • 測試案例
  • 測試計劃
  • 使用者手冊
  • 驗收標準

不包含：

  • 加密程式碼（無法理解）
  • 第三方執行檔（無法檢視）

審查的目的

1. 防止誤解

  • 確保理解一致
  • 減少認知差異

2. 增加理解

  • 團隊成員共同學習
  • 知識分享

3. 防止缺陷流入下階段

  • 早期發現問題
  • 避免連鎖反應

07 7.5 審查流程活動 ⭐⭐⭐ 必考

五個流程活動（順序固定）

記憶：規啟個審修

1. 規劃（Planning）

定義：定義審查目標與範圍

主要工作：

  • 定義審查對象
  • 決定審查範圍
  • 安排時間與人力
  • 定義引入/引出標準

負責人：

  • 審查負責人
  • 管理者

引入標準（Entry Criteria）：

什麼條件下可以進入審查

例子：

✓ 作者已自我檢查
✓ 文件格式正確
✓ 相關資源已準備

引出標準（Exit Criteria）：

什麼條件下審查結束

例子：

✓ 所有 Issue 已記錄
✓ 所有爭議已解決
✓ 審查報告已完成

2. 審查者啟動（Review Initiation）

定義：將工作產品交給審查者

主要工作：

  • 分發審查文件
  • 提供輔助資訊
  • 說明審查重點
  • 回答審查者疑問

輔助資訊：

  • 查核表（Checklist）
  • 審查指引
  • 相關規範
  • 背景說明

重要性：

讓審查者有工具可用

3. 個人審查（Individual Review）

定義：審查者獨立審查工作產品

性質：

  • 非必要活動
  • 某些審查類型可省略

主要工作：

  • 仔細審查文件
  • 記錄問題於 Issue Log
  • 標記疑慮之處

Issue Log vs Bug Report

為何叫 Issue Log？

原因：

很明顯的錯誤 → 記錄
看不懂的地方 → 也記錄（帶到會議釐清）
可能的問題 → 記錄（待確認）

不是所有 Issue 都是 Bug：

  • 有些是誤解
  • 有些需釐清
  • 有些確實是 Bug

重要性：

認真審查的關鍵活動

4. 審查會議/溝通與分析（Communication and Analysis）

定義：討論並確認 Issue

主要工作：

  • 彙總所有 Issue Log
  • 逐一討論
  • 確認是否為缺陷
  • 指派修復者
  • 決定優先順序

為何需要會議：

審查者 A：
  這裡我看不懂（Issue）
    ↓ 會議中
  作者解釋
    ↓ 結果
  原來是我誤解（非 Bug）

審查者 B：
  這裡有錯（Issue）
    ↓ 會議中
  大家確認
    ↓ 結果
  確實是 Bug

會議產出：

  • 確認的缺陷清單
  • 修復責任分配
  • 優先順序排定

5. 修復與跟催（Fixing and Reporting）

定義：修正缺陷並確認完成

主要工作：

修復階段：

  • 作者修正缺陷
  • 回報修復狀態

跟催階段：

  • 審查者確認修復
  • 決定是否接受
  • 必要時要求重修

度量指標收集：

  • 審查時間
  • 發現缺陷數
  • 修復時間
  • 用於持續改善

流程順序 ⭐⭐⭐ 必考

完整順序：

  1. 規劃
  2. 審查者啟動
  3. 個人審查
  4. 溝通與分析
  5. 修復與跟催

注意：

  • 個人審查是非必要
  • 其他 4 個通常都要執行

考試重點：

  • 會考順序
  • 監控不在順序中（隨時進行）

08 7.6 審查角色 ⭐⭐⭐ 必考

六種審查角色

1. 管理者（Manager）

職責：

  • 出人
  • 出錢
  • 出資源

特性：

  • 不建議參加審查會議
  • 提供支援即可

為何不建議參加：

原因1：可能阻礙自由討論
  - 有老闆在，大家不敢暢所欲言

原因2：技術審查與管理者脫節
  - 技術細節管理者未必懂
  - 參加效益不高

2. 審查負責人（Review Leader）

職責：

  • 負責行政事務
  • 安排會議
  • 跟催進度
  • 確保審查完成

特性：

  • 類似專案經理角色
  • 確保流程順利

3. 主持人（Facilitator/Moderator）⭐

職責：

  • 確保會議有效進行
  • 準時開始、結束
  • 每個議題都有結論
  • 調停爭議
  • 保持客觀

重要性：

有效審查的關鍵角色

考題常考：

哪個角色是有效審查的關鍵？ 答案：主持人

4. 審查者（Reviewer）

職責：

  • 審查工作產品
  • 提出問題
  • 客觀評論

要求：

  • 具備專業性
  • 保持客觀

重要原則：

評論針對工作產品，不針對作者

錯誤示範：

✗ 「你寫得真爛」
✓ 「這裡的邏輯可以改善」

5. 作者（Author）

職責：

  • 解釋工作產品
  • 接受建議
  • 進行修復

態度：

  • 開放心胸
  • 接受批評
  • 虛心改進

特性：

  • 工作產品的創作者
  • 最了解內容
  • 負責修正

6. 記錄者（Scribe）⭐

職責：

  • 公正記錄會議內容
  • 記錄所有反饋
  • 記錄所有問題

重要性：

確保資訊完整傳達

為何需要獨立記錄者：

如果作者記錄：

作者心態：
  這個建議太難改 → 忘記記錄
  這個可以改 → 記錄下來
    ↓ 結果
  記錄不完整

獨立記錄者：

記錄者：
  客觀記錄所有內容
  不受個人偏好影響
    ↓ 結果
  記錄完整公正

角色階梯圖 📊

正式程度由高到低：

檢閱（Inspection）：
  管理者 ✓
  審查負責人 ✓
  主持人 ✓
  記錄者 ✓
  審查者（多位）✓
  作者 ✓

技術審查（Technical Review）：
  審查負責人 ✓
  主持人 ✓
  記錄者 ✓
  審查者（多位）✓
  作者 ✓

演練導覽（Walkthrough）：
  記錄者 ✓
  審查者（1位以上）✓
  作者 ✓

非正式審查（Informal Review）：
  審查者（1位）✓
  作者 ✓

09 7.7 審查類型 ⭐⭐⭐ 必考

四種審查類型（按正式程度排序）

記憶順序：非演技檢

正式程度的判斷標準

三個維度：

1. 角色完整性

  • 6 個角色都有 → 很正式
  • 只有 2 個角色 → 不正式
  • 角色可兼任 → 較不正式

2. 流程執行度

  • 5 個流程都跑 → 很正式
  • 只跑 2-3 個流程 → 較不正式
  • 幾乎沒跑流程 → 非正式

3. 文件產出

  • 大量文件 → 很正式
  • 少量文件 → 較不正式
  • 無文件 → 非正式

類型1：非正式審查（Informal Review）

特性

最不正式

流程：

  • 幾乎沒有流程
  • 快速簡便

目的：

  • 快速發現異常
  • 得到即時回饋

文件：

  • 不需要文件

時機：

  • 隨時可進行
  • 常見於敏捷開發

例子

場景1：請同事幫忙看

開發者：「你可以幫我看一下這段程式碼嗎？」
同事：「好啊」
  ↓ 2分鐘後
同事：「這裡有個問題」
開發者：「謝謝，我馬上改」

場景2：吃飯聊天

午餐時間
  ↓
「你的需求我看不太懂」
「哪裡？我說明一下」
  ↓ 釐清
問題解決

特點：

  • 沒有正式會議
  • 沒有文件產出
  • 快速有效

類型2：演練導覽（Walkthrough）⭐

特性

作者主導 ⭐ 必考

目的：

  • 教育訓練
  • 達成共識
  • 讓大家理解工作產品

個人審查：

  • 非必要（可選）

比喻：這堂課程

作者：老師（創作講義） 審查者：學生（學習講義） 目的：讓學生理解講義內容

過程：

第1天：
  老師帶領學生理解講義
  學生提出疑問
  老師解釋說明
    ↓
第2天：
  繼續深入講解
  釐清不清楚的地方
  達成共識
    ↓ 結果
  大家對講義內容有一致理解

執行方式

1. 作者解釋

  • 逐頁講解
  • 說明設計理念
  • 回答問題

2. 審查者參與

  • 提出疑問
  • 給予建議
  • 發現問題

3. 達成共識

  • 共同理解
  • 一致認知

適用情境

適合：

  • 新團隊成員
  • 複雜設計
  • 需要教育訓練

不適合：

  • 時間緊迫
  • 內容簡單
  • 團隊已熟悉

類型3：技術審查（Technical Review）⭐

特性

技術導向

目的：

  • 評估技術品質
  • 提出解決方案
  • 給予替代方案

個人審查：

  • 必要

為何個人審查必要？

原因：

技術審查需要：
  - 思考解決方案
  - 評估替代方案
  - 技術可行性分析
    ↓
會議當下很難臨場發揮
  ↓ 所以
需要事先審查、思考

主持人的重要性

為何需要主持人：⭐

技術討論容易：

方案 A 支持者：「A 方案比較好」
方案 B 支持者：「B 方案更適合」
  ↓ 沒有主持人
爭論不休
  ↓ 有主持人
「我們先聽 A 的理由，再聽 B 的理由，然後投票決定」

結論：

技術審查一定要有主持人

特色

1. 專業審查者

  • 技術背景
  • 領域專家

2. 提出方案

  • 不只找問題
  • 要給解決方法

3. 可能多種方案

  • 各有優缺點
  • 需要討論決定

類型4：檢閱（Inspection）

特性

最正式

流程：

  • 5 個流程活動都要執行

角色：

  • 6 個角色都要有
  • 各自獨立，不兼任

文件：

  • 大量文件產出
  • 詳細記錄

度量指標收集 ⭐

特有活動：

  • 收集審查數據
  • 分析審查效果
  • 持續改善

收集項目：

  • 審查時間
  • 發現缺陷數
  • 缺陷類型
  • 修復時間

目的：

  • 衡量審查有效性
  • 改善審查流程

適用情境

必要情況：

  • 高風險專案
  • 需要認證
  • 法規要求
  • 關鍵系統

例子：

✓ 飛機控制系統
✓ 醫療設備軟體
✓ 金融交易系統
✓ 核電廠控制系統

特性：

  • 最花時間
  • 最花成本
  • 最嚴謹
  • 留有證據

11 7.9 審查類型的選擇

選擇依據

1. SDLC 類型

  • 敏捷開發 → 非正式審查
  • 瀑布式開發 → 正式審查

2. 流程成熟度

  • 成熟組織 → 正式審查
  • 新興團隊 → 非正式審查

3. 工作產品複雜度

  • 複雜 → 正式審查
  • 簡單 → 非正式審查

4. 法規要求

  • 有要求 → 檢閱
  • 無要求 → 依需求選擇

5. 時間壓力

  • 時間緊 → 非正式
  • 時間充裕 → 正式
