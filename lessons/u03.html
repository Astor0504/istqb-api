01 單元3：軟體測試七大原則

02 學習目標

  • 理解並記憶軟體測試的七大原則
  • 能夠辨識實際情境對應的測試原則
  • 理解每個原則背後的實務意義
  • 將原則應用於測試實踐中

03 3.1 為什麼需要測試原則？

測試原則的重要性

  • 指導測試活動的方向
  • 避免常見的測試誤區
  • 提供決策的依據
  • 建立共同的測試認知

考試重點

題型：透過情境描述，判斷對應哪個測試原則

難度：K2（理解層級）

策略：

  • 不要死背文字
  • 理解原則精神
  • 記住關鍵例子

04 3.2 原則1：測試顯示缺陷的存在，而非不存在

英文名稱

Testing shows the presence, not the absence of defects

釣蝦池理論 🎣

情境

在釣蝦池釣蝦：

  • 釣到蝦 → 證明池裡有蝦 ✓
  • 釣不到蝦 → 不能證明池裡沒蝦 ✗

可能原因

釣不到蝦的原因：

  1. 技術太差
  2. 位置不對
  3. 時間不對
  4. 方法不對

測試的對應

測試找到 Bug    →  證明軟體有問題 ✓
測試沒找到 Bug  →  不能證明軟體沒問題 ✗

可能原因：

  1. 測試方向錯誤
  2. 測試不夠深入
  3. Bug 隱藏很深
  4. 測試技術不適合

實務意義

對測試人員

  • 不要保證「絕對沒問題」
  • 只能說「目前測試範圍內沒發現問題」
  • 要持續改進測試方法

對利害關係人

  • 理解測試的極限
  • 零缺陷是不可能的
  • 要接受殘留風險

測試的真正目標

雖然無法證明零缺陷，但測試可以：

  • 降低軟體中殘留未發現缺陷的機率
  • 找出大部分的缺陷
  • 找出最關鍵的缺陷
  • 建立對品質的信心

05 3.3 原則2：窮舉測試是不可行的

英文名稱

Exhaustive testing is impossible

什麼是窮舉測試？

定義

測試所有可能的輸入組合

填單考卷的例子

  • 30 道選擇題
  • 每題 4 個選項
  • 所有可能性 = 4³⁰ = 1,152,921,504,606,846,976 種

結論：即使超級電腦也無法在合理時間內完成

Chrome 瀏覽器設定的例子

情境

某個設定頁面：

  • 40 個「2選1」選項
  • 1 個「3選1」選項
  • 所有可能性 = 2⁴⁰ × 3 = 3,298,534,883,328 種

問題：

  1. 測試完這個頁面需要多久？
  2. Chrome 還有很多其他頁面和功能
  3. 全部測試是天文數字

為什麼窮舉測試不可行？

三大限制

  1. 時間不夠：組合數量太龐大
  2. 金錢不夠：成本無法負擔
  3. 人力不夠：沒有足夠資源

解決方案：選擇性測試

策略

  1. 聚焦測試（Focused Testing）
  • 測試最重要的功能
  • 測試最常用的路徑
  1. 風險導向測試（Risk-Based Testing）
  • 優先測試高風險區域
  • 依風險排序測試優先級
  1. 使用測試技術
  • 等價劃分：找代表性的值
  • 邊界值分析：測試邊界
  • 決策表：測試規則組合

測試技術的威力

Chrome 設定例子

  • 窮舉：3,298,534,883,328 種組合
  • 使用組合測試技術：約 6 種組合就夠

差距：從 3 兆降到 6（選擇性測試）

取捨

✓ 優點：大幅降低測試案例數量 ✗ 缺點：必須承擔風險（未測到的組合可能有問題）

06 3.4 原則3：早期測試節省時間與金錢

英文名稱

Early testing saves time and money

修復成本倍增

歷史數據（2002年）

  • 需求階段：成本 = 1
  • 發佈階段：成本 = 30

更新數據（2010年）

  • 需求階段：成本 = 1
  • 發佈階段：成本 = 100

現代預估（2025年）

  • 需求階段：成本 = 1
  • 發佈階段：成本 = 200+ （推估）

原因：軟體越來越複雜（AI、行動裝置、物聯網）

為什麼早期測試成本低？

文件 vs 程式碼

修改文件（需求階段）
  ↓ 容易
  只要改文字

修改程式碼（開發階段）
  ↓ 較困難  
  需要改程式、重新編譯、測試

修改已發佈系統（發佈後）
  ↓ 最困難
  改程式 + 測試 + 部署 + 客戶溝通

邏輯：

  • 文件亂寫 → 程式碼亂開發 → 測試亂設計
  • 文件正確 → 程式碼正確 → 測試正確

測試左移（Shift Left）

傳統思維

需求 → 設計 → 開發 → [測試]
                    ↑
                測試在這裡才開始

測試左移

[測試] → [測試] → [測試] → [測試]
  ↓        ↓        ↓        ↓
需求 ←→ 設計 ←→ 開發 ←→ 上線

作法：

  • 需求階段：審查需求文件
  • 設計階段：審查設計規格
  • 開發階段：靜態分析、單元測試
  • 整合階段：整合測試
  • 上線前：系統測試、驗收測試

測試左移的挑戰

組織層面

  1. 需要支持：利害關係人願意投入
  2. 需要成本：額外的時間與人力
  3. 需要共識：團隊理解價值

證明價值

老闆會問：「投入測試左移，有什麼好處？」

數據說話：

  • 早期發現 1 個缺陷 = 省 30-100 倍成本
  • 減少後期返工
  • 提高上市成功率

07 3.5 原則4：缺陷的群集效應

英文名稱

Defects cluster together

帕雷托法則（80/20法則）

定義

20% 的原因造成 80% 的結果

來源：義大利經濟學家 Vilfredo Pareto

  • 發現：20% 的富人擁有 80% 的財富
  • 應用：各個領域

在軟體測試的應用

  • 20% 的功能模組包含 80% 的缺陷
  • 20% 的程式碼產生 80% 的問題

缺陷分佈圖

均勻分佈（理想但不真實）

功能A  功能B  功能C  功能D
 🐛🐛   🐛🐛   🐛🐛   🐛🐛

每個功能缺陷數量相同（不符合現實）

群集分佈（實際情況）

功能A  功能B  功能C  功能D
 🐛     🐛🐛🐛  🐛     🐛🐛🐛🐛🐛
             🐛🐛🐛

缺陷集中在某些功能（符合現實）

如何應用此原則？

1. 識別群集區域

可能指標：

  • 程式碼複雜度高
  • 修改頻繁的模組
  • 新功能區域
  • 開發者經驗不足的部分
  • 過去有問題的模組

2. 調整測試資源

策略：

高缺陷密度區 → 投入 60% 測試資源
中缺陷密度區 → 投入 30% 測試資源  
低缺陷密度區 → 投入 10% 測試資源

而非平均分配（25% / 25% / 25% / 25%）

3. 持續監控

做法：

  • 收集缺陷數據
  • 分析分佈模式
  • 動態調整資源

學生曠課的例子 📚

  • 20% 的學生佔了 80% 的曠課次數
  • 曠課的總是那幾個
  • 對應測試：有問題的總是那幾個模組

08 3.6 原則5：測試會失效（殺蟲劑效應）

英文名稱

Tests wear out (Pesticide Paradox)

殺蟲劑效應的由來 🪳

小強（蟑螂）的抗藥性

  1. 第一次用殺蟲劑 → 小強死亡 ✓
  2. 重複使用同款殺蟲劑 → 小強產生抗藥性
  3. 持續使用同款殺蟲劑 → 小強免疫，殺不死 ✗

解決方法：定期更換殺蟲劑品牌

測試的殺蟲劑效應

情境

設計 10 個測試案例 → 重複執行 → 都通過
                    ↓
              價值逐漸降低

為什麼價值降低？

  • 這 10 個測試案例是針對性設計
  • 針對已知缺陷設計
  • 重複執行找不到新缺陷

結果

Day 1: 🐛🐛🐛🐛🐛 → 找到 5 個 Bug
Day 2: 🐛🐛 → 找到 2 個 Bug  
Day 3: 🐛 → 找到 1 個 Bug
Day 4: - → 找不到新 Bug（但系統可能還有問題）

解決方案

1. 定期更新測試案例

  • 新增：加入新的測試案例
  • 修改：調整測試數據
  • 刪除：移除過時的測試案例

2. 變化測試數據

案例：

原始測試：age = 25, name = "John"
變化測試：age = 65, name = "李明"
         age = 10, name = "小華"
         age = 100, name = "王老先生"

3. 新增異常情境測試

原本只測：正常登入 新增測試：

  • 錯誤密碼
  • 空白輸入
  • SQL Injection
  • 特殊字元

例外：可以重複執行的測試

核心功能測試

  • 公司的核心業務流程
  • 無論改版幾次都不能錯
  • 這類測試案例必須重複執行

例子：

  • 銀行轉帳功能
  • 電商結帳流程
  • 登入驗證

迴歸測試（Regression Testing）

  • 確保修改沒有破壞原有功能
  • 必須重複執行相同測試
  • 建議自動化

效能測試

  • 長期運行、重複執行
  • 監控系統穩定性
  • 找出記憶體洩漏等問題

本原則的精神

不要期待用同樣的測試找到新問題 要定期檢視並更新測試策略

09 3.7 原則6：測試依情境而異

英文名稱

Testing is context dependent

核心觀念

不能用一種測試方法測試所有系統

不同系統，不同重點

電子商務系統

重點：

  • ✓ 可用性（Usability）：好不好用
  • ✓ 效能（Performance）：速度快不快
  • ✓ 安全性（Security）：交易安全
  • ✓ 兼容性（Compatibility）：多平台支援

案例：網頁載入速度

  • 2017 年標準：5 秒內
  • 現代標準：0.5 秒內
  • 超過標準 → 使用者流失

國防系統

重點：

  • ✓ 安全性（Security）：最優先
  • ✓ 可靠性（Reliability）：不能當機
  • ✓ 正確性（Correctness）：計算精準
  • 相對次要：使用者介面美觀

醫療系統

重點：

  • ✓ 安全性：人命關天
  • ✓ 正確性：診斷精準
  • ✓ 可靠性：不能失誤
  • ✓ 法規遵循：符合醫療標準

遊戲軟體

重點：

  • ✓ 娛樂性：好不好玩
  • ✓ 效能：不能 LAG
  • ✓ 相容性：多平台
  • 相對次要：百分百正確性（可容忍小 Bug）

瀏覽器的差異

Chrome vs IE：

  • Chrome 支援最新 Web 標準
  • IE 支援度較低
  • 測試重點不同

結論：即使同為瀏覽器，測試方法也不同

測試需要的能力

知識與技能

  • 瞭解該領域的特性
  • 知道關鍵品質要求
  • 熟悉常見問題模式

判斷力

  • 評估風險
  • 選擇測試技術
  • 安排測試優先順序

創造力

  • 設計異常測試案例
  • 找出隱藏的缺陷
  • 模擬真實使用情境

10 3.8 原則7：無缺陷的謬論

英文名稱

Absence-of-defects fallacy

核心觀念

零缺陷 ≠ 產品成功

經典案例：《我的英雄夢》遊戲

背景

  • 公司：智冠港龍（台灣遊戲公司）
  • 靈感：寶可夢（Pokémon GO）很紅
  • 想法：做台灣版，抓宮廟神明

開發過程

  • 投入大量開發與測試資源
  • 測試完整，沒發現嚴重 Bug
  • 產品品質良好

結果

  • ❌ 市場反應冷淡
  • ❌ 下載量極低
  • ❌ 商業失敗

原因分析

產品本身沒問題，但是：

  • 市場定位錯誤
  • 使用者不感興趣
  • 競爭力不足
  • 沒有解決使用者痛點

啟示

完美的測試 → 零缺陷 → 產品品質好
                         ↓
                      但不保證
                         ↓
                    產品會成功

成功的條件

必要條件

  1. 產品有價值：解決使用者問題
  2. 市場需求：有人願意用
  3. 商業模式：能夠獲利

品質只是其中一環

產品成功 = 品質 + 市場 + 時機 + 行銷 + 價格...

對測試人員的啟示

不只關注缺陷

  • 也要關心：產品是否符合使用者期望
  • 驗證（Verification）：做對了嗎？
  • 確認（Validation）：做對的東西嗎？

與業務代表合作

  • 了解商業目標
  • 理解使用者需求
  • 參與驗收測試

12 3.10 原則的實務應用

案例1：新功能開發

情境

電商網站新增「推薦系統」功能

應用原則

原則6（依情境而異）：

  • 這是電商系統 → 重視可用性、速度
  • 推薦系統 → 還要重視準確性

原則4（群集效應）：

  • 推薦演算法複雜 → 可能有較多缺陷
  • 投入更多測試資源

原則3（早期測試）：

  • 需求階段：審查推薦邏輯
  • 設計階段：審查演算法設計
  • 開發階段：單元測試、整合測試

原則2（窮舉不可行）：

  • 用戶數百萬，商品上萬種
  • 無法測試所有組合
  • 使用抽樣測試、邊界值分析

原則5（測試失效）：

  • 定期更新測試數據
  • 新增邊緣案例
  • 測試不同用戶輪廓

原則1（顯示存在）：

  • 不保證零缺陷
  • 提供風險評估報告

原則7（無缺陷謬論）：

  • 確認推薦真的有用（驗收測試）
  • 收集使用者回饋

案例2：Legacy 系統維護

情境

維護 10 年前開發的銀行系統

應用原則

原則5（測試失效）：

  • 10 年的測試案例可能過時
  • 需要全面檢視更新

原則4（群集效應）：

  • 舊問題集中區域要特別注意
  • 新修改的部分也可能產生群集

原則3（早期測試）：

  • 變更需求時就開始審查
  • 不要等到全部改完才測試

原則2（窮舉不可行）：

  • 系統龐大，無法全部重測
  • 使用影響分析選擇測試範圍

13 3.11 考試重點整理

題型分析

K1 題型（記憶）

問法：「以下何者為測試七大原則？」

策略：記住原則名稱

K2 題型（理解）✨ 最常考

問法：「以下情境對應哪個測試原則？」

範例題目：

情境：測試團隊發現即使持續執行相同的測試案例，
      也無法找到新的缺陷。

問題：此情境對應哪個測試原則？

答案：原則5（測試會失效/殺蟲劑效應）

考試陷阱

陷阱1：混淆相似原則

  • 原則1（顯示存在） vs 原則7（無缺陷謬論）
  • 前者：測試的限制
  • 後者：產品成功的條件

陷阱2：過度聯想

錯誤思考：

早期測試省錢 → 所以要窮舉測試？

正確理解：

早期測試 + 選擇性測試（因為窮舉不可行）

15 3.13 互動練習

練習1：原則配對

題目：將左側情境與右側原則配對

情境

A. 團隊發現 80% 的缺陷集中在登入模組 B. 無法測試年齡欄位的所有可能值 C. 相同測試執行多次後不再發現新問題 D. 產品測試全部通過但市場反應冷淡 E. 在需求階段發現問題比開發階段便宜

原則

  1. 原則3：早期測試
  2. 原則4：群集效應
  3. 原則5：測試失效
  4. 原則2：窮舉不可行
  5. 原則7：無缺陷謬論

答案：A-2, B-4, C-3, D-5, E-1

練習2：情境分析

情境： 你負責測試一個新的行動支付 App。專案時程緊迫， 資源有限。請運用測試原則規劃測試策略。

思考方向：

  1. 原則2：無法測試所有組合 → 決定用什麼測試技術？
  2. 原則6：這是金融 App → 什麼品質特徵最重要？
  3. 原則4：哪些功能最容易有問題 → 如何分配資源？
  4. 原則3：如何儘早開始測試？
  5. 原則5：如何確保測試案例持續有效？

練習3：選擇題

題目： 某測試團隊針對一個新開發的系統進行測試，發現即使 執行了完整的測試案例，系統仍然沒有發現任何缺陷。 測試經理因此向管理層報告「系統已完美無缺」。

問題：這違反了哪個測試原則？
