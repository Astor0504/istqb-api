03 2.1 什麼是軟體測試？

官方定義

軟體測試是一個規劃、準備和評估的過程，用以確定元件或系統及相關工作產品符合特定需求、適合使用目的，並能發現缺陷。

白話解釋

測試就是了解軟體品質現況的方法，透過各種手段來：

  • 發現軟體的問題
  • 降低軟體失敗的風險
  • 提供資訊給利害關係人做決策

測試不是什麼

❌ 不是證明軟體完全沒有問題（不可能做到） ❌ 不是開發人員的責任（是獨立的專業） ❌ 不是為了刁難開發人員（是為了品質）

04 2.2 測試 vs 除錯 vs 品質保證

三者關係圖

┌─────────────────────────────────────┐
│     品質保證（QA）                    │
│  Quality Assurance                   │
│  - 預防導向                           │
│  - 流程改善                           │
│  - 建立標準                           │
└─────────────────────────────────────┘
         │
         ↓
┌─────────────────────────────────────┐
│     品質管控（QC）                    │
│  Quality Control                     │
│  - 檢測導向                           │
│  - 發現問題                           │
│  - 產品檢驗                           │
│         │                             │
│         ├→ 測試（Testing）            │
│         └→ 審查（Review）             │
└─────────────────────────────────────┘
         │
         ↓
┌─────────────────────────────────────┐
│     除錯（Debugging）                 │
│  - 開發人員執行                       │
│  - 找出缺陷原因                       │
│  - 修復缺陷                           │
└─────────────────────────────────────┘

詳細比較

項目測試（Testing）除錯（Debugging）品質保證（QA）
執行者測試人員開發人員品保人員/流程專家
目的發現缺陷、失效修復缺陷預防缺陷
時機整個SDLC發現缺陷後專案全程
方法動態/靜態測試程式碼分析流程改善
產出測試報告修復的程式碼流程文件

重要觀念

  • QA（品質保證）：照著標準做，就會成功（預防）
  • QC（品質管控）：檢查做出來的東西（檢測）
  • 測試是 QC 的一部分，不是 QA

05 2.3 核心術語：錯誤、缺陷、失效

三者的因果關係

人為錯誤（Error）
    ↓
產生缺陷（Defect/Bug）
    ↓
導致失效（Failure）

詳細說明

1. 錯誤（Error）

定義：人的行為產生不正確的結果

案例：

  • 程式設計師在吵雜環境下誤解需求
  • 設計師因為疲累而設計錯誤的介面
  • 需求分析師遺漏重要需求

根本原因（Root Cause）：

  • 電話鈴聲干擾 → 程式設計師分心 → 產生錯誤
  • 時間壓力 → 程式設計師疲累 → 產生錯誤

2. 缺陷（Defect/Bug）

定義：工作產品中不符合需求的部分

案例：

  • 程式碼中邊界值判斷錯誤
  • 文件中的不一致描述
  • 設計中的邏輯錯誤

重要：

  • Defect = Bug（同義詞）
  • 可透過靜態測試直接發現
  • 可透過動態測試間接發現（透過失效）

3. 失效（Failure）

定義：系統無法執行應有功能的表現

案例：

  • 計算機顯示 1+2+3=24（錯誤結果）
  • 系統當機（無法運作）
  • 登入按鈕無反應（功能失效）

特性：

  • 只能在系統執行時被觀察到
  • 不是所有缺陷都會導致失效
  • 失效是缺陷的外在表現

實例分析

案例：火箭爆炸事件（Ariane 5）

  1. 錯誤（Error）：程式設計師直接複製貼上舊版程式碼，未考慮新火箭速度更快
  2. 缺陷（Defect）：程式碼中的整數溢位（Integer Overflow）
  3. 失效（Failure）：69秒後火箭爆炸

教訓：

  • 複製貼上前應該審查程式碼
  • 靜態測試可以提早發現這類缺陷
  • 動態測試難度高（火箭發射成本極高）

06 2.4 測試的必要性

軟體失敗的四大影響

1. 💰 金錢損失

  • 企業投入大量資金開發
  • 推出劣質產品導致損失
  • 需要重工修復（成本倍增）

案例：

  • 火箭爆炸：數十億歐元損失
  • 系統當機：營業損失

2. ⏰ 時程延誤

  • 品質問題導致無法準時上市
  • 錯失市場先機
  • 競爭對手搶佔市場

案例：

  • Beta 版問題太多，上市延遲
  • 修復缺陷耗時，專案延宕

3. ⚠️ 人身安全

  • 醫療系統誤判：病人死亡
  • 交通系統故障：飛機失事、捷運事故
  • 軍事系統錯誤：安全威脅

真實案例：

  • 名古屋空難：飛行系統與機長搶奪控制權，200+ 人死亡
  • 醫療放射線：軟體誤判癌細胞數量，給予錯誤劑量
  • 手機爆炸：軟體漏洞被遠端控制

4. 😞 使用者不滿意度

  • App 評分過低，無人下載
  • 使用體驗差，用戶流失
  • 口碑不佳，影響品牌

07 2.5 測試的目標

七大測試目標

1. 評估工作產品（Evaluate Work Products）

範圍：不只測試軟體，還包括：

  • 需求文件
  • 使用者故事（User Story）
  • 設計規格
  • 測試案例

2. 發現缺陷（Find Defects）

方式：

  • 靜態測試：直接發現缺陷
  • 動態測試：透過失效證明缺陷存在

3. 引發失效（Trigger Failures）

目的：讓潛在問題浮現 方法：設計測試案例刻意觸發

4. 了解覆蓋範圍（Provide Coverage）

目標：知道測試的完整程度 公式：覆蓋率 = 已測項目 / 總項目

5. 降低風險（Reduce Risk）

作用：降低劣質軟體的風險等級 範圍：產品風險（不含專案風險）

6. 驗證與確認（Verification & Validation）

驗證（Verification）：

  • 問題：「做對了嗎？」（Are we building it right?）
  • 檢查：是否符合規格
  • 方法：靜態測試（審查）

確認（Validation）：

  • 問題：「做對的東西嗎？」（Are we building the right thing?）
  • 檢查：是否符合使用者需求
  • 方法：動態測試

7. 提供資訊給利害關係人

資訊包括：

  • 測試進度
  • 發現的缺陷數量
  • 品質現況
  • 風險評估
  • 是否建議上線

08 2.6 測試目標會因情境而異

影響因素

工作產品複雜度

  • 簡單系統 → 測試目標較單純
  • 複雜系統 → 需要更全面的測試目標

測試層級

  • 單元測試目標：觸發失效、儘早發現問題
  • 驗收測試目標：確認系統可運作、符合商業需求

識別的風險

  • 高風險 → 加強測試
  • 低風險 → 簡化測試

開發模型（SDLC）

  • 瀑布式 → 測試在後期
  • 敏捷式 → 測試貫穿全程

其他因素

  • 公司文化
  • 競爭壓力
  • 上市時程
  • 法規要求

09 2.7 測試案例的來源：正確答案在哪裡？

釣蝦池理論

在釣蝦池裡釣到蝦 → 證明池裡有蝦 在釣蝦池裡釣不到蝦 → 不能證明池裡沒蝦（可能是技術不好）

對應測試：

  • 測試找到 Bug → 證明軟體有問題 ✓
  • 測試沒找到 Bug → 不能證明軟體沒問題 ✗

正確答案的來源

計算機範例

問題：1 + 2 + 3 = ?

  • 你知道正確答案是 6
  • 計算機顯示 24 → 你知道有問題
  • 因為你知道正確答案

軟體測試的正確答案

來源順序：

  1. 規格書（最主要）
  2. 使用者故事
  3. 需求文件
  4. 設計文件
  5. 客戶的期望（在客戶腦袋裡）

重要觀念：

  • 正確答案不是測試人員說的
  • 正確答案不是開發人員說的
  • 正確答案在測試依據（Test Basis）

10 2.8 測試依據（Test Basis）

定義

任何可以幫助說明受測系統運作的資訊或文件

常見的測試依據

  • 功能需求規格
  • 使用案例（Use Case）
  • 使用者故事（User Story）
  • 商業規則
  • 程式碼（Code）
  • 系統架構圖
  • 資料流程圖

測試依據的重要性

測試依據完整 → 測試設計容易 → 測試品質高
    ↓
測試依據殘缺 → 測試設計困難 → 測試品質差

黑箱測試特別依賴文件完整性：

  • 文件好 → 容易設計測試案例
  • 文件爛 → 無法設計測試案例
  • 文件沒有 → 改用經驗導向測試

12 2.10 真實案例分析

案例1：魔獸世界瘟疫事件

事件經過

  1. 設計：副本中的 Boss「哈卡」有技能「墮落之血」（可傳染）
  2. 預期：瘟疫只存在於副本內
  3. 缺陷：獵人的寵物中毒後，收回寵物再釋放，瘟疫被帶出副本
  4. 失效：主城大規模感染，玩家、NPC 大量死亡，伺服器過載

測試觀點

正常情境：

  • 玩家進入副本
  • 玩家中毒
  • 玩家離開副本 → 瘟疫應消失 ✓

異常情境（未測到）：

  • 寵物中毒
  • 收回寵物
  • 離開副本
  • 釋放寵物 → 瘟疫外洩 ✗

教訓：

  • 正常情境與異常情境都要測試
  • 未測到的情境就是風險
  • 使用者的玩法可能超出想像

案例2：高鐵訂票系統

問題

  • 一站車票原本 15 元
  • 系統顯示 1000+ 元

原因分析

可能的缺陷：

  1. 計算邏輯錯誤
  2. 資料庫異常
  3. 價格參數設定錯誤

測試不足：

  • 未測試各站間價格計算
  • 未測試價格顯示正確性
  • 未做邊界值測試

案例3：好市多重複結帳

個人經驗

  • 藥品區結帳 → 結帳一次
  • 推到櫃台 → 又結帳一次
  • 結果：同樣商品被收費兩次

測試觀點

測試項目：

  • [ ] 藥品區結帳流程
  • [ ] 櫃台結帳流程
  • [ ] 重複結帳檢查（未測到）

教訓：沒測試就不知道有問題

13 2.11 正常情境 vs 異常情境

定義

正常情境（Normal/Positive Testing）

  • 定義：依照規格書描述的使用方式
  • 來源：需求文件、規格書
  • 特性：有明確定義
  • 舉例：正確帳號密碼登入成功

異常情境（Abnormal/Negative Testing）

  • 定義：非預期的使用方式
  • 來源：測試人員經驗與想像
  • 特性：難以窮舉
  • 舉例：惡意輸入、極端值、異常操作

測試設計策略

原則：

  • ✓ 正常情境：用最少測試案例覆蓋
  • ✓ 異常情境：用一對一方式測試

理由：

  • 異常情境做一對一 → 失敗時容易定位問題
  • 正常情境最小化 → 節省測試成本

14 2.12 測試的價值與限制

測試的價值

早期發現問題

  • 需求階段發現 → 修復成本 = 1
  • 發佈階段發現 → 修復成本 = 30-100 倍

降低修復成本曲線

修復成本
   │
高 │                              ╱
   │                          ╱
   │                      ╱
   │                  ╱
   │              ╱
   │          ╱
低 │──────╱──────────────────────→
   需求 設計 開發 測試 發佈   時間

結論：越早測試，成本越低

測試的限制

測試原則1：測試無法證明零缺陷

  • 可以證明：缺陷存在
  • 無法證明：缺陷不存在

原因：

  • 測試方向可能錯誤
  • 缺陷隱藏很深
  • 測試不可能窮舉

實際目標

  • 找出最多的缺陷
  • 找出最嚴重的缺陷
  • 降低殘留缺陷的風險

15 2.13 動態測試 vs 靜態測試

比較表

項目動態測試靜態測試
時機系統可執行時系統無法執行前
方法執行程式審查、工具掃描
發現失效（Failure）缺陷（Defect）
階段SDLC 中後期SDLC 早期
成本修復成本高修復成本低
技術黑箱、白箱測試審查、靜態分析

動態測試（Dynamic Testing）

定義：需要執行軟體的測試

特性：

  • 系統必須能運作
  • 通常在開發中後期
  • 透過失效找到缺陷

技術：

  • 黑箱測試（功能測試）
  • 白箱測試（結構測試）
  • 經驗導向測試

靜態測試（Static Testing）

定義：不執行軟體的測試

特性：

  • 可在早期執行
  • 直接發現缺陷
  • 修復成本低

技術：

  1. 審查（Review）：人工檢查
  • 需求審查
  • 設計審查
  • 程式碼審查
  1. 靜態分析（Static Analysis）：工具掃描
  • 程式碼複雜度
  • 編碼規範檢查
  • 安全漏洞掃描

測試左移（Shift Left）

概念：測試應該儘早開始

傳統做法：
需求 → 設計 → 開發 → [測試開始]

測試左移：
[測試開始] 需求 → 設計 → 開發 → 測試

好處：

  • 早期發現問題
  • 修復成本低
  • 避免錯誤累積

17 2.15 本單元總結

核心觀念

  1. ✓ 測試是為了發現缺陷、降低風險
  2. ✓ 測試≠除錯≠品質保證（三者不同）
  3. ✓ 錯誤→缺陷→失效（因果關係）
  4. ✓ 正確答案在測試依據，不在測試人員
  5. ✓ 動態測試與靜態測試互補
  6. ✓ 測試左移可降低成本

必背內容

  • [ ] ISO 29119（軟體測試標準）
  • [ ] ISO 25010（品質特徵標準）
