第1章 · 測試基礎

ISTQB認證介紹與考試準備

ISTQB 是全球最大的軟體測試認證組織，在 120+ 國家都認可

ISTQB認證介紹與考試準備

Foundation Level 是所有進階證照的必備基礎，證書終身有效

ISTQB認證介紹與考試準備

考試：40題選擇題，60分鐘（繁中版），及格需答對 26 題（65%）

ISTQB認證介紹與考試準備

證照層級：Foundation（基礎）→ Advanced（進階）→ Expert（專家）

軟體測試基礎概念

測試的目的：發現缺陷、降低風險、提供品質資訊——不是「證明沒問題」

軟體測試基礎概念

QA（品質保證）= 預防導向，改善流程；QC（品質管控）= 檢測導向，找問題；測試是 QC 的一部分

軟體測試基礎概念

Error → Defect → Failure 因果鏈：人犯錯(Error) → 程式有缺陷(Defect) → 執行時失效(Failure)

軟體測試基礎概念

Verification（驗證）= 做對了嗎？符合規格；Validation（確認）= 做對的東西嗎？符合需求

軟體測試基礎概念

靜態測試：不執行程式碼就能找問題（審查文件）；動態測試：執行程式碼才能找問題

軟體測試七大原則

① 測試顯示缺陷存在，不能證明沒有缺陷（釣蝦池：釣不到蝦不代表沒蝦）

軟體測試七大原則

② 窮舉測試不可行（不可能測所有組合，要選擇有代表性的）

軟體測試七大原則

③ 早期測試節省時間和金錢（越晚發現，修復越貴）

軟體測試七大原則

④ 缺陷群集效應：80% 的問題藏在 20% 的模組（帕雷托法則）

軟體測試七大原則

⑤ 測試會磨損（Tests wear out）：同樣的測試反覆做，就找不到新蟲了，要定期更新測試

軟體測試七大原則

⑥ 測試情境依賴：電商和核電廠的測試方式不同

軟體測試七大原則

⑦ 無缺陷謬論：零缺陷 ≠ 好產品，要測試符合使用者需求

測試活動與產物

7大測試活動順序：規劃 → 分析 → 設計 → 建置 → 執行 → 監控（持續進行）→ 完成

測試活動與產物

注意：「監控」是貫穿全程的，不在排序選項中！

測試活動與產物

規劃：產出測試計劃書；分析：產出測試條件；設計：產出測試案例；建置：準備測試環境和資料

測試活動與產物

執行：產出測試日誌和缺陷報告；完成：產出測試總結報告

測試活動與產物

可追溯性（Traceability）：把需求、測試案例、缺陷串在一起，方便追蹤影響範圍

測試活動與產物

兩種角色：測試管理者（管規劃、資源）vs 測試人員（做分析、設計、執行）

測試人員的技能與好的實踐

測試通用技能：測試知識、溝通能力、分析思維、細心謹慎、懷疑精神

測試人員的技能與好的實踐

全團隊方法（Whole Team Approach）：開發、測試、業務全員負責品質，不只是測試人員的事

測試人員的技能與好的實踐

測試獨立性優點：避免作者盲點、更客觀；缺點：溝通成本高、可能造成瓶頸

測試人員的技能與好的實踐

獨立性層次：自己測自己的 < 團隊互測 < 獨立測試部門 < 外部測試

第2章 · 測試與開發

軟體開發生命週期與測試層級

SDLC 三大類別：循序式（Sequential）、迭代式（Iterative）、增量式（Incremental）——瀑布式是循序式的一種

軟體開發生命週期與測試層級

5大測試層級：① 單元測試（開發人員）② 元件整合測試（開發人員）③ 系統測試（測試人員）④ 系統整合測試（測試人員）⑤ 驗收測試（使用者）

軟體開發生命週期與測試層級

V 模型：每個開發階段對應一個測試層級，形成 V 字形，是循序式的代表

軟體開發生命週期與測試層級

TDD（測試驅動開發）：先寫測試再寫程式碼

軟體開發生命週期與測試層級

BDD（行為驅動開發）：用 Given/When/Then 格式描述行為

軟體開發生命週期與測試層級

DevOps：開發和維運整合，強調快速回饋和持續整合（CI/CD）

軟體開發生命週期與測試層級

Shift Left（測試左移）：越早測試越省成本

驗收測試與測試類型

4種驗收測試：UAT（使用者驗收）、操作驗收（IT維運）、合約驗收（符合合約）、法規驗收（符合法律）

驗收測試與測試類型

功能性測試：測「做什麼」（登入功能、計算功能）

驗收測試與測試類型

非功能性測試：測「做得多好」（速度、安全、易用性）

驗收測試與測試類型

確認性測試（Confirmation Testing）：Bug 修完後重新測試確認修好了

驗收測試與測試類型

迴歸測試（Regression Testing）：改了某個地方，確保沒有破壞其他地方

驗收測試與測試類型

維護測試（Maintenance Testing）：系統上線後要修改或環境改變時執行

第3章 · 靜態測試

靜態測試技術

靜態測試 = 不執行程式碼（看文件、看程式碼）；動態測試 = 執行程式碼

靜態測試技術

靜態分析：用工具自動掃描程式碼（找死碼、未使用變數等）

靜態測試技術

審查（Review）的 5 個流程：規劃 → 審查者啟動 → 個人審查 → 溝通與分析 → 修復與跟催

靜態測試技術

口訣：「規啟個溝修」

靜態測試技術

審查的 6 種角色：發起者、作者、主持人、審查員、記錄員、管理者

靜態測試技術

4種審查類型（正式度由低到高）：非正式審查 → 演練導覽（作者主導）→ 技術審查 → 檢閱

靜態測試技術

口訣：「非演技檢」

靜態測試技術

審查成功因素：明確目標、合適類型、相關人員參與、客觀態度（非針對人）、管理層支持

第4章 · 動態測試技術

動態測試技術總覽：黑箱測試

三大測試技術類型：黑箱（Black-box）、白箱（White-box）、經驗導向（Experience-based）

動態測試技術總覽：黑箱測試

黑箱測試：只看「輸入什麼、輸出什麼」，不管程式碼內部怎麼寫

動態測試技術總覽：黑箱測試

依據：規格文件；適合所有人（不需懂程式碼）

動態測試技術總覽：黑箱測試

黑箱測試技術包含：等價劃分、邊界值分析、決策表、狀態轉換

動態測試技術總覽：黑箱測試

覆蓋率（Coverage）：測試了多少比例，越高代表測試越完整

等價劃分測試技術（EP）

核心概念：同一組的資料處理方式相同，所以只需測一個代表值就夠了

等價劃分測試技術（EP）

分類：有效等價類別（Valid）= 程式應接受的輸入；無效等價類別（Invalid）= 程式應拒絕的輸入

等價劃分測試技術（EP）

一個連續範圍通常分成 3 類：小於下限（無效）、範圍內（有效）、大於上限（無效）

等價劃分測試技術（EP）

覆蓋率公式：已測試的類別數 / 總類別數 × 100%

等價劃分測試技術（EP）

重要：同一等價類別裡的多個值，只算覆蓋 1 個類別！

邊界值分析測試技術（BVA）

核心想法：程式碼的 Bug 最常出現在「邊界」（例如 ≥ 寫成 >）

邊界值分析測試技術（BVA）

二值法（2-value BVA）：每個邊界取 2 個值 = 邊界本身 + 相鄰分區最近的值

邊界值分析測試技術（BVA）

三值法（3-value BVA）：每個邊界取 3 個值 = 邊界-1、邊界本身、邊界+1

邊界值分析測試技術（BVA）

三值法比二值法更嚴格，可以發現更多邊界錯誤

邊界值分析測試技術（BVA）

範例：範圍 0-10，精確度 1；二值法選 -1、0、10、11；三值法再加 1、9

決策表測試技術

適用時機：有多個條件、不同組合產生不同結果（例如：滿額折扣、會員等級）

決策表測試技術

規則數 = 各條件可能值的乘積（若全是 True/False，則是 2^N，N = 條件數量）

決策表測試技術

決策規則數量和「結果（動作）有幾個」完全無關！

決策表測試技術

覆蓋率 = 已測規則數 / 總規則數 × 100%

決策表測試技術

精簡決策表：把相同結果的規則合併，減少測試案例數

狀態轉換測試技術

適用時機：系統有明確的狀態（如：已登入/未登入、鎖定/解鎖）

狀態轉換測試技術

狀態圖：圓圈代表狀態，箭頭代表轉換（觸發條件 / 動作）

狀態轉換測試技術

狀態表：把狀態圖轉成表格形式，更容易設計測試案例

狀態轉換測試技術

覆蓋標準（3種）：狀態覆蓋 < 有效轉換覆蓋（0-switch）< 所有轉換覆蓋

狀態轉換測試技術

0-switch = 測試所有有效的直接轉換

狀態轉換測試技術

N-switch = 測試 N+1 個連續轉換的序列（1-switch 就是兩個轉換的連續序列）

白箱測試技術

白箱測試 = 需要看程式碼，測試「如何實作的」

白箱測試技術

敘述覆蓋（Statement Coverage）= 每行程式碼都執行到的比例

白箱測試技術

分支覆蓋（Branch Coverage）= 每個 if/else 的 True 和 False 分支都執行到的比例

白箱測試技術

包含關係：分支覆蓋 > 敘述覆蓋。100% 分支覆蓋 → 保證 100% 敘述覆蓋（反之不成立）

白箱測試技術

白箱測試的限制：無法發現「功能沒有實作」的問題（根本沒寫那段程式碼）

白箱測試技術

v4.0.1 用語：Branch Testing（不是 Decision Testing）

經驗導向測試技術

錯誤推測（Error Guessing）：根據以前的經驗猜測哪裡容易出錯，然後針對那裡測試

經驗導向測試技術

探索性測試（Exploratory Testing）：邊學習、邊設計、邊執行，適合規格不完整時

經驗導向測試技術

以 Session 為基礎：設定一個時間盒（如90分鐘），在這段時間內自由探索

經驗導向測試技術

測試章程（Test Charter）：探索性測試的「任務說明書」，描述要探索什麼

經驗導向測試技術

查表測試（Checklist-based Testing）：根據清單逐一測試，確保不遺漏

經驗導向測試技術

最適合：規格不完整、時間緊迫、測試人員有豐富經驗的情境

第5章 · 測試管理

測試估計技術

基於度量指標的技術：比率估計（用歷史數據的比例）、推估法（從當前數據外插）

測試估計技術

基於專家的技術：廣義德菲法（多位專家面對面多輪估計）、三點估計

測試估計技術

三點估計公式：E = (a + 4m + b) / 6；記憶：1-4-1 除以 6

測試估計技術

a = 最樂觀；m = 最可能；b = 最悲觀

測試估計技術

規劃撲克（Planning Poker）：廣義德菲法在敏捷的變形，用費氏數列牌

測試估計技術

絕對估計（小時/天）vs 相對估計（故事點數）

測試案例排序策略

三種排序策略：基於風險（高風險先測）、基於覆蓋率（高覆蓋先測）、基於需求重要性

測試案例排序策略

最重要原則：依賴關係 > 優先性！不滿足依賴就無法執行，所以依賴永遠優先

測試案例排序策略

例如：TC2 需要 TC1 先跑（TC1 創建帳號，TC2 才能登入），不管 TC2 優先性再高也要先跑 TC1

測試案例排序策略

解題技巧：選項回推法——從答案選項的第一個位置開始驗證，排除不可能的選項

風險管理基礎

風險 = 發生機率 × 影響程度（兩者缺一不可）

風險管理基礎

產品風險（Product Risk）= 與品質有關（安全漏洞、功能錯誤、效能差），測試可以舒緩

風險管理基礎

專案風險（Project Risk）= 與管理有關（預算超支、人員流失、進度延誤），測試不能解決

風險管理基礎

四種風險應對策略：接受（不處理）、轉移（買保險）、應變計劃（備案）、舒緩（降低機率/影響）

風險管理基礎

測試的角色 = 舒緩產品風險

風險管理基礎

風險導向測試：高風險 → 投入更多測試資源；低風險 → 投入較少

測試監控與報告

測試監控 = 監督（收集資料）+ 控制（採取行動改善）

測試監控與報告

測試進度報告（定期產出）：目前狀態、執行進度、缺陷統計、阻礙事項、下一步計劃

測試監控與報告

測試總結報告（階段結束後）：整體摘要、差異分析、缺陷統計、剩餘風險、經驗教訓

測試監控與報告

關鍵差異：「經驗教訓」在總結報告；「阻礙事項和下一步計劃」在進度報告

測試監控與報告

引入標準（Entry Criteria）= 開始測試的前提條件（環境就緒、文件完成）

測試監控與報告

引出標準（Exit Criteria）= 結束測試的條件（覆蓋率達標、高優先缺陷修復完）

測試監控與報告

缺陷管理：缺陷報告的10個重要欄位，其中重現步驟是最關鍵的

測試工具與自動化

工具支援測試的方式：提高效率、減少人為錯誤、快速回饋、易於收集指標

測試工具與自動化

測試自動化的效益：節省重複測試時間、提高一致性、可以在非工作時間執行

測試工具與自動化

測試自動化的風險：初期投資高、維護成本大、不切實際的期待（買了就成功的迷思）

測試工具與自動化

最適合自動化：迴歸測試（重複執行、穩定、可預測）

測試工具與自動化

不適合自動化：探索性測試、可用性測試（需要人的判斷力）

測試工具與自動化

構型管理（Configuration Management）：確保測試人員使用正確的版本
